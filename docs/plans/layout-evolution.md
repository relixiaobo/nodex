# Side Panel 布局方案演进与对比

> 2026-03-23 — 四种方案的回顾与新方向

## 背景

soma 作为 Chrome Side Panel 应用（300-700px 宽），核心体验是 **Outliner（笔记）+ AI Chat（对话）** 的协同。四次布局迭代本质上都在解决同一个问题：**如何在窄空间里让用户在笔记和 AI 之间高效切换？**

---

## 方案 A：多面板并排（PanelLayout）

> 已废弃。PR #142 之前的原始设计。

```
┌──────────┬──────────┐
│ Panel 1  │ Panel 2  │   宽屏：并排卡片
│ (Node)   │ (Node)   │
└──────────┴──────────┘

┌──────────────────────┐
│ [Notes ▾]       👤   │   窄屏：下拉切换
├──────────────────────┤
│ Active Panel         │
└──────────────────────┘
```

**心智模型**：N 个独立面板，像浏览器 tab。Chat 是其中一个面板。

**优点**：
- 灵活——可以同时打开多个节点
- 宽屏体验好——并排比较

**缺点**：
- 窄屏退化严重——面板挤在一起，下拉切换很重
- Chat 和 Node 是平级的——AI 没有特殊地位，很容易被埋在多个面板里
- 状态管理复杂——N 个面板的历史、焦点、展开状态
- 心智冲突——用户已经有浏览器 tab，再在侧边栏做一层 tab 太重

---

## 方案 B：Chat 底层 + Panel 浮层（DeskLayout v3）

> 已废弃。PR #142 的设计。

```
┌──────────────────────┐
│ [Notes ▾]  ✦   👤   │
├──────────────────────┤
│ Node Panel (浮层)     │   面板盖在 Chat 上面
│                      │
├──────────────────────┤
│ Chat (底层，被遮挡)    │   点击 ✦ 收起面板，露出 Chat
└──────────────────────┘
```

**心智模型**：Chat 是桌面（底层环境），NodePanel 是纸张（浮层焦点）。

**优点**：
- 层级清晰——笔记在上，AI 在下
- 改动最小——只改 4 个文件

**缺点**：
- 窄屏 Chat 完全被遮挡——需要收起面板才能看到
- "收起面板"的交互不直觉——用户不知道 Chat 在面板下面
- Chat 打开时笔记完全不可见——跟方案 A 一样是二选一

---

## 方案 C：双视角 Toggle（ToggleLayout）

> 当前已实现（PR #171）。

```
┌─────────────┬──────────────┬──┐
│ 💬 Chat     │ 📋 Outliner  │👤│
├─────────────┴──────────────┴──┤
│                               │
│  当前活跃视角全屏               │
│                               │
└───────────────────────────────┘
```

**心智模型**：Chat 和 Outliner 是同一工作空间的两个视角，一键切换。

**优点**：
- 最简单——只有两个状态，一目了然
- 状态保持好——两个视图始终 mounted，切换不丢状态
- Chrome tab 风格——用户熟悉的交互
- 标签栏提供常驻导航——总能看到另一个视角

**缺点**：
- **仍然是二选一**——看笔记就看不到 AI，跟 AI 对话就看不到笔记
- 切换频繁——"边看笔记边问 AI"的场景需要反复切换
- 上下文丢失感——切到 Chat 后忘了刚才在看哪个节点
- 标签栏占用垂直空间——Side Panel 寸土寸金

---

## 方案 D：Outliner 底层 + Chat 抽屉（新提案）

> 待实现。

```
默认状态（Outliner 全屏 + 悬浮输入框）：
┌──────────────────────────┐
│  面包屑  W / ... / Today  │
│  # day                   │
│  Today, Sun, Mar 23      │
│                          │
│  • 节点 1                 │
│  • 节点 2                 │
│                          │
│  ┌──────────────────────┐│
│  │ 💬 Ask your notes... │││  ← 悬浮输入框
│  └──────────────────────┘│
└──────────────────────────┘

发送消息后（Chat 抽屉从底部滑出）：
┌──────────────────────────┐
│  面包屑（可见）            │  ← Outliner 顶部露出
│░░░░░░░░░░░░░░░░░░░░░░░░░░│  ← 半透明遮罩
│┌────────────────────────┐│
││  ▼ 下拉收起             ││  ← 抽屉拖拽区
││                        ││
││  AI: 分析如下...         ││
││                        ││
││  ┌────────────────────┐││
││  │ 继续对话...         │││
││  └────────────────────┘││
│└────────────────────────┘│
└──────────────────────────┘
```

**心智模型**：Outliner 是你的工作台（始终在），AI 是你的助手（需要时呼出）。

**优点**：
- **笔记始终可见**——不需要切换就能看到笔记上下文
- **最少交互**——输入框常驻，打字就能对话，不需要先"切换到 Chat"
- **层级自然**——笔记是主体，AI 是辅助层，符合"笔记是焦点，AI 是环境"
- **渐进式**——轻量问题不打开抽屉（未来可做 inline 回答），深度对话打开抽屉
- **节省空间**——没有标签栏，输入框复用为 Chat 入口
- **上下文不丢失**——抽屉打开时仍能看到 Outliner 顶部（面包屑），知道自己在哪

**缺点**：
- 抽屉遮挡 Outliner——深度对话时仍然看不到完整笔记
- Chat 历史入口不明显——怎么回到之前的对话？
- 抽屉交互复杂度——高度控制、拖拽、关闭方式需要设计
- Chat 不再是一等公民——纯 Chat 场景（不看笔记）体验可能退化

## 对比矩阵

| 维度 | A 多面板 | B Chat 底层 | C Toggle | D Chat 抽屉 |
|------|---------|------------|----------|------------|
| **笔记+AI 同时可见** | ⚠️ 宽屏可以 | ❌ | ❌ | ⚠️ 抽屉打开时部分可见 |
| **切换成本** | 高（找面板） | 中（收起面板） | 低（一键） | **最低（直接输入）** |
| **心智负担** | 高（N 面板） | 中（层级概念） | 低（二选一） | **最低（只有笔记+输入框）** |
| **状态管理** | 复杂 | 中等 | 简单 | 简单 |
| **窄屏适配** | 差 | 差 | **好** | **好** |
| **AI 入口** | 不明显 | 不明显 | 明确（标签） | **最明确（常驻输入框）** |
| **纯 Chat 场景** | 中 | 中 | **好（全屏 Chat）** | ⚠️ 需要设计 |
| **空间利用** | 差（标签+面板） | 中 | 中（标签栏） | **好（无标签栏）** |
| **实现复杂度** | 高 | 低 | 低 | 中 |

## 方案 D 的关键设计问题

### 1. 输入框设计

悬浮在 Outliner 底部，类似 iOS 的 home indicator 区域：
- 单行输入框 + 发送按钮
- 点击或输入文字 → 展开为完整输入区域（多行 + 模型选择器）
- 发送后 → 抽屉滑出

### 2. 抽屉行为

- **打开**：从底部滑出，覆盖 Outliner 的 80% 高度
- **关闭**：下拉拖拽 / 点击遮罩 / 点击 Outliner 露出的面包屑区域
- **高度**：默认 80%，可拖拽调整（记住偏好）
- **Outliner 露出**：顶部始终露出面包屑（约 40px），让用户知道自己在哪个节点

### 3. Chat 历史

- 抽屉内顶部有会话标题（可编辑）
- 下拉可切换到历史会话列表
- 或者在 ⌘K 命令面板中搜索历史对话

### 4. 从 Chat 引用节点

- Chat 中点击 `<ref>` → 关闭抽屉，Outliner 导航到该节点
- Chat 中的 `<node />` 内嵌 → 仍然可用（抽屉内看节点内容）

### 5. 纯 Chat 场景

- 抽屉可以拖到 100% 高度（全屏 Chat）
- 或者保留 Toggle 标签作为备选入口

---

## 建议

方案 D 最符合 soma 的产品定位——**笔记工具 + AI 辅助**，而不是 **AI 聊天工具 + 笔记功能**。

核心判断：用户 80% 的时间在看/写笔记，20% 的时间跟 AI 对话。方案 D 让 80% 的场景（笔记）零成本，20% 的场景（AI）一次输入即达。而方案 C（Toggle）让两个场景都需要显式切换。

实施建议：在 ToggleLayout 基础上迭代，保留 ChatPanel / NodePanel 组件不变，只改布局层。
