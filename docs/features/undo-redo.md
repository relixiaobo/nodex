# Feature: 撤销与重做

> Phase 2 | 文本撤销已有（TipTap 内置），导航撤销已有，节点操作撤销未实现

## 概述

用户通过 `Cmd+Z` 撤销上一步操作，`Cmd+Shift+Z` 重做被撤销的操作。撤销系统覆盖三个层次：文本编辑、节点操作、导航。三者通过统一的优先级机制无缝衔接。

## 当前实现状态

| 层次 | 状态 | 说明 |
|------|------|------|
| 文本编辑撤销 | ✅ 已有 | TipTap StarterKit 内置 ProseMirror History |
| 导航撤销 | ✅ 已有 | `navUndoStack` / `navRedoStack`，`use-nav-undo-keyboard.ts` |
| 节点操作撤销 | ❌ 未实现 | 创建/删除/移动等操作不可撤销 |
| 三层统一优先级 | ❌ 未实现 | 当前 Cmd+Z 在编辑模式走 TipTap，非编辑模式走 navUndo |

## 架构设计

### 三层撤销栈

```
┌─────────────────────────────────────────────────────┐
│  Cmd+Z 按下                                          │
│                                                      │
│  1. 编辑器内？→ TipTap History.undo()                 │
│     └ TipTap 有可撤销步骤？→ 执行文本撤销              │
│     └ TipTap 无可撤销步骤？→ 继续到 2                  │
│                                                      │
│  2. 节点操作栈有内容？→ nodeOpsUndo()                  │
│     └ 撤销最近的节点操作                               │
│     └ 栈为空？→ 继续到 3                               │
│                                                      │
│  3. 导航栈有内容？→ navUndo()                          │
│     └ 撤销最近的导航                                   │
│     └ 栈为空？→ 无操作                                 │
└─────────────────────────────────────────────────────┘
```

### 优先级规则

当用户按下 `Cmd+Z` 时：

1. **先撤销文本编辑** — 如果当前节点的 TipTap 编辑器有未撤销的文字或格式变更，优先回退文本
2. **再撤销节点操作** — 文本编辑历史用完后，回退结构性操作（创建、删除、移动等）
3. **最后撤销导航** — 节点操作历史也用完后，回退面板导航

三者无缝衔接，用户无需感知层次切换。

## 行为规格

### 文本编辑撤销

- 所有在节点中输入的文字变更、格式变更（加粗、斜体、删除线、代码、高亮）均可撤销
- **连续快速打字视为一次操作** — 撤销时不逐字回退，而是回退到上一次停顿前的状态
  - TipTap/ProseMirror 默认实现：按输入时间自动分组
- 仅当 TipTap 编辑器处于挂载状态时有效
- 失焦（`onBlur`）后文本已持久化到 store，TipTap History 在下次聚焦时重置

### 节点操作撤销 — 未实现

#### 可撤销的操作

| 类别 | 操作 |
|------|------|
| 创建 | 回车新建节点、拆分节点 |
| 删除 | 删除空节点（Backspace）、合并节点、移入回收站 |
| 移动 | 缩进（Tab）、反缩进（Shift+Tab）、上移（Cmd+Shift+↑）、下移（Cmd+Shift+↓） |
| 拖拽 | 拖拽排序到新位置 |
| 状态 | 切换完成状态（checkbox） |
| 引用 | 创建节点引用、创建行内引用、引用类型互转 |
| 标签 | 应用标签、创建并应用标签、移除标签 |
| 字段 | 应用字段、创建并应用字段、删除字段 |
| 批量 | 以上所有操作的多选批量版本 |

#### 不可撤销的操作

| 操作 | 原因 |
|------|------|
| 永久删除（清空回收站） | 数据库层面不可逆 |
| Supabase 同步冲突解决 | 服务端合并结果 |

#### 操作记录格式

```typescript
interface UndoEntry {
  type: string;                  // 操作类型标识
  timestamp: number;             // 记录时间
  focusState: {                  // 操作时的焦点状态（用于恢复）
    focusedNodeId: string | null;
    focusedParentId: string | null;
    cursorPos?: number;          // TipTap 光标位置（如可获取）
  };
  forward: () => void;          // 重做函数
  backward: () => void;         // 撤销函数
}
```

#### 撤销实现策略

**方案：命令模式（Command Pattern）**

每个可撤销操作在执行时记录足够的信息以生成逆操作：

| 操作 | 记录 | 撤销动作 |
|------|------|---------|
| 创建节点 | `{ parentId, nodeId, position }` | 删除该节点 |
| 删除节点 | `{ parentId, node snapshot, position, children snapshots }` | 在原位置恢复节点及子树 |
| 缩进节点 | `{ nodeId, oldParentId, oldPosition, newParentId }` | 移回原父节点原位置 |
| 反缩进节点 | 同上 | 移回原父节点原位置 |
| 移动节点 | `{ nodeId, oldParentId, oldPosition, newParentId, newPosition }` | 移回原位置 |
| 应用标签 | `{ nodeId, tagDefId, createdTupleIds }` | 移除标签 + 删除 created tuples |
| 移除标签 | `{ nodeId, tagDefId, tupleSnapshots }` | 重新应用标签 + 恢复 tuples |
| 应用字段 | `{ nodeId, tupleId, attrDefId }` | 移除字段 |

**关键约束**：
- 创建标签并应用到节点 = 一步操作。撤销时移除标签应用关系，但标签定义（tagDef）保留（可能已被其他节点使用）
- 同理，创建字段并应用 = 一步。撤销时移除字段应用，attrDef 保留

### 导航撤销

当前已实现。`navUndoStack` / `navRedoStack` 存储 `{ panelHistory, panelIndex }` 快照。

- `navigateTo()` 自动 push 快照到 undo 栈，清空 redo 栈
- `navUndo()` → 恢复上一个导航快照
- `navRedo()` → 恢复下一个导航快照
- Session-only，不持久化到 storage

### 焦点恢复

每次撤销或重做后，光标自动回到该操作发生时的位置：

- 记录操作时的 `focusedNodeId` + `focusedParentId`
- 撤销后恢复 focus 状态
- 如原节点已不存在（被其他操作删除），聚焦到最近的可见兄弟节点

### 历史管理规则

| 规则 | 说明 |
|------|------|
| 新操作清空 redo | 执行新操作后，被撤销的操作不可再重做 |
| 最大步数 | 最多保留 100 步节点操作历史 |
| 快速连续 Cmd+Z | 逐步回退，不跳过或重复 |
| 展开/折叠不记录 | 视图状态变更不是数据操作 |
| 空操作不记录 | 操作结果与原状态相同时不入栈 |
| 撤销失败 | 跳过该步（如节点已被远端删除），其余历史正常可用 |

### 特殊场景

| 场景 | 行为 |
|------|------|
| 创建标签并应用到节点 | 一步操作。撤销时移除标签关系，tagDef 保留 |
| 撤销 Zoom 导航 | 回到导航前的视图位置（navUndo 处理） |
| 展开/折叠未改变状态 | 不记录 |
| 文本编辑结果与原文相同 | 不记录（TipTap 自动处理） |
| 撤销失败 | 跳过，其余历史正常 |

## 实现考量

### Zustand + Supabase 撤销的挑战

当前所有节点操作通过 `node-store.ts` 直接写入 Supabase。撤销需要：

1. **乐观更新已实现** — store 的 immer mutations 是同步的，Supabase 写入是异步的
2. **撤销 = 逆向乐观更新 + 逆向 Supabase 写入**
3. **冲突风险** — 撤销时如果其他客户端已修改了同一节点，可能冲突。建议：
   - 单用户阶段（当前）：直接覆盖
   - 多用户阶段：检查 version 号，冲突时提示用户

### TipTap History 集成点

TipTap 的 History extension 需要暴露"是否有可撤销步骤"给外层：

```typescript
// 查询 TipTap 是否可撤销
editor.can().undo()  // → boolean
```

当 `editor.can().undo()` 返回 `false` 时，Cmd+Z 事件应传递到节点操作层。

### 与多选批量操作的关系

批量操作（如选中 5 个节点后按 Delete）应作为**单步**入栈，撤销时一次性恢复所有节点。

## 决策记录

| 日期 | 决策 | 原因 |
|------|------|------|
| 2026-02-14 | 三层优先级：文本 → 节点操作 → 导航 | 与用户直觉一致：先撤销最近的细粒度操作，再撤销粗粒度操作 |
| 2026-02-14 | 使用 Command Pattern（记录正/逆函数） | 比全量快照更节省内存；操作种类有限，逆操作可枚举 |
| 2026-02-14 | 节点操作撤销栈 100 步上限 | 平衡内存开销与实用性 |
| 2026-02-14 | 展开/折叠不入撤销栈 | 视图偏好不是数据变更，频繁展开折叠会污染撤销历史 |
| 2026-02-14 | 创建标签/字段并应用 = 一步撤销 | 用户感知为单一动作；撤销时 tagDef/attrDef 保留以防被其他节点引用 |
| 2026-02-14 | 节点操作撤销定为 Phase 2 | 当前优先完善 Phase 1 功能（日期节点、标签/字段增强），撤销系统是体验增强 |
