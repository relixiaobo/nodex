## 残留问题：cleanupHashTagText 文本删除可能失败

`handleHashTagSelect` → `cleanupHashTagText()` 的流程：

```typescript
const cleanupHashTagText = useCallback(() => {
    const ed = editorRef.current;
    if (!ed) { /* fallback: strip from store name */ return; }

    const text = ed.getText();
    const caret = ed.getCaretOffset() ?? text.length;
    const range = findHashTriggerRange(text, caret)
      ?? (hashRangeRef.current.from < hashRangeRef.current.to ? hashRangeRef.current : null);

    if (range) {
      ed.deleteTextRange(range);  // ← 这里可能失败
    }
    // ...
}, ...);
```

### 可能失败点

1. **`ed.getCaretOffset()` 返回 null 或错误位置**: Enter 按下时浏览器可能已移动焦点，导致 caret 不在 `#per` 后面 → `findHashTriggerRange` 找不到
2. **`ed.deleteTextRange(range)` DOM 操作失败**: `createRangeFromTextOffsets` 返回 null → textContent fallback 也可能失败
3. **React 重渲染覆盖 DOM 修改**: `deleteTextRange` 直接修改 DOM，但如果 React 在同一帧重渲染（如 `applyTag` 触发 store 更新），DOM 修改可能被覆盖

### 建议诊断

`cleanupHashTagText` 中已有 `[DBG-hashCleanup]` 和 `[DBG-hashCleanup-after]` 日志。需要用户再次复现，确认这两个日志是否出现，以及 `afterText` 是否确实删除了 `#per`。

如果 `afterText` 仍包含 `#per`，则 `deleteTextRange` 的 DOM 操作有问题。
如果 `afterText` 已不含 `#per` 但视觉上仍显示，则 React 重渲染覆盖了 DOM。

## 修复方案（02-14 已合入）

**根因**: `cleanupHashTagText` 中 `ed.deleteTextRange(range)` 的 DOM 操作可能因范围计算错误而静默失败，`#per` 文本未被删除。

**修复**: 在 `deleteTextRange` 调用后增加安全检查：读回 `ed.getText()`，如果仍然匹配 `#trigger` 模式，则使用 `ed.setPlainText()` 进行 store 级别的强制清理。

```typescript
const afterText = ed.getText();
const stillHasHash = afterText.match(/#([^\s#@]*)$/u);
if (stillHasHash) {
  const cleaned = afterText.replace(/#([^\s#@]*)$/u, '');
  ed.setPlainText(cleaned, cleaned.length);
  if (userId) updateNodeName(nodeId, ed.getHTML(), userId);
} else {
  if (userId) updateNodeName(nodeId, afterHtml, userId);
}
```
