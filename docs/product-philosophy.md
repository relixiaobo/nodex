# soma 产品方法论 v1

> 记下来，用得上。

---

## 一、我们相信什么

### 笔记的唯一价值是"有用"

如果记下来的东西在未来用不上，没有人会坚持记录。工具的职责不是让用户记更多笔记，而是让每一条笔记都有更高的概率在未来被用上。

笔记有用的前提有两个：
- **笔记本身被消化过** — 用自己的话写、经过组织和连接，而非原文复制粘贴
- **在需要的时候能出现** — 不依赖用户记住"我记过这个"，而是在正确的时刻主动浮现

### 笔记是对信息的预处理

记笔记不是存档，而是思考。把信息用自己的话重新表述、分类、连接的过程，本身就是理解和记忆的过程。不经过预处理的信息，未来大概率用不上——要么想不起来，要么找到了也需要重新理解一遍。

不要 think later，要 think now。

### 工具应该让人变强，而不是让人觉得工具很强

解构外部信息（总结、翻译、提取），用户会觉得"工具很强"。帮助用户建构自己的想法（组织、连接、应用），用户才会觉得"我变强了"。soma 追求后者。

## 二、方法论：记下来，用得上

soma 的知识工作流是一个三步循环：

```
捕获（在浏览中记录）
  ↓
消化（在大纲中组织）
  ↓
重遇（在需要时浮现）
  ↓
  └──→ 强化记录的动力 ──→ 捕获更多 ...
```

### 第一步：捕获 — 在哪里阅读，就在哪里思考

> "务必立即记下乍现的灵光与感动，一旦错失，便难以重觅。"

现代人的信息源头是浏览器。soma 活在浏览器侧边栏里，让用户在阅读的同时记录，不需要切换应用、不需要思考"该放哪里"。

捕获的摩擦越低，记录的频率越高：
- **笔记即记录** — 高亮不是笔记，用自己的话写下的 note 才是。高亮只是触发思考的上下文
- **剪藏即结构** — 网页内容自动带上来源、日期、标签
- **IM 即入口** — 在聊天窗口发一条消息，就是一条笔记
- **日记即默认** — 所有捕获先进入 Today，不强迫分类

#### 高亮与笔记的关系

高亮必须伴随用户自己写的 note 才能保存。这是一个刻意的设计约束：

```
#source（网页）
 └── 用户的想法 #note          ← 主体：自己的话
      ├── Highlight:: 原文高亮  ← 上下文：触发思考的原文
      ├── 进一步展开...
      └── ...
```

**笔记是主体，高亮是证据。** 如果你写不出一句自己的话来回应一段高亮，说明它还没有和你的经验、问题、目标产生真正的连接——认同和触动不等于有用，放过它。

这个约束也让未来的回顾和 AI 洞察更有价值——分析的是你自己的想法，而不是别人文章的片段。

### 第二步：消化 — 组织就是理解

> "组织笔记的过程带来了理解的必要难度，也许思考才是最重要的记笔记的原因。"

捕获解决"别丢了"，但信息不会自动变成知识。消化是将碎片变成自己的东西的过程。soma 用大纲作为消化的介质——大纲的每一个操作都是一种认知行为：

| 操作 | 认知行为 |
|------|---------|
| 把 A 拖到 B 下面 | "A 属于 B" — 分类 |
| 调整兄弟节点顺序 | "这个比那个重要" — 优先级判断 |
| 给节点打标签 | "这是一个 X" — 定义和归类 |
| 给节点加字段 | "它的属性是 Y" — 结构化 |
| 折叠一组子节点 | "这部分我理解了" — 聚焦 |
| 创建引用链接 | "这和那个有关" — 建立连接 |
| 删除一条笔记 | "这对我没用" — 筛选和提纯 |

大纲的优势在于：**消化和记录用同一个界面**。不需要从"笔记模式"切换到"思考模式"。你在日记里看到昨天剪藏的内容，把它拖到一个主题节点下，加上标签，调整位置——这个过程看起来只是"整理"，实际上你已经完成了理解。

消化的关键准则：
- **用自己的话写** — 核心是先理解，再记录，而不是鹦鹉学舌
- **把删笔记作为日常动作** — 认同和触动不等于有用，只保留有用的
- **标签从简单开始** — 不要设计完美体系，让标签随使用自然生长

### 第三步：重遇 — 在正确的时刻出现

> "旧笔记往往没准备好被使用——需要理解当时的上下文，找到跟新事情的联系，代价比收益高，不如找新的。"

这是所有笔记工具的死亡之谷：记了 → 沉底 → 再也没看到 → 觉得白记 → 不记了。

soma 用三条路径打破这个循环：

| 路径 | 触发方式 | 用户感知 |
|------|---------|---------|
| **浏览器重遇** | 用户打开与笔记相关的网页 | "我之前记过这个" |
| **IM 推送** | soma 基于时间/事件主动推送 | "这条旧笔记今天有用" |
| **对话问答** | 用户向 soma 提问 | "我之前关于 X 记过什么？" |

浏览器重遇是 soma 独有的优势——**用户什么都不用做，只是继续上网，笔记自己出现了**。反馈延迟取决于用户的浏览频率，而不是用户的自律程度。

IM 渠道扩展了触达场景：用户不需要打开浏览器也能和自己的知识互动——问问题、收推送、随手记。

## 三、正反馈循环：产品成败的关键

> "付出小于回报时，人们就会坚持；付出大于回报时，人们就会放弃。" — 岩田聪

soma 的正反馈循环：

```
记录（低付出）──→ 笔记在未来被用上（高回报）──→ 觉得记录有价值 ──→ 记录更多
```

这个循环要转起来，需要满足三个条件：

1. **记录的付出足够低** — 高亮一下、发条消息，3 秒完成
2. **回报来得足够快** — 不是三个月后回顾时才发现有用，而是下次浏览相关网页时就出现
3. **回报感足够强** — 不是"这条笔记还在"，而是"这条笔记帮我解决了当下的问题"

### 冷启动问题

新用户刚开始没有笔记，没有笔记就没有重遇，没有重遇就感受不到价值。因此新用户引导的核心目标不是教功能，而是**尽快让用户完成第一次"记了 → 用上了"的体验**。

## 四、soma 的独特位置

### 为什么是浏览器侧边栏？

四个维度上，soma 做到了独立应用做不到的事：

| 维度 | 独立应用的痛点 | soma 的浏览器优势 |
|------|--------------|------------------|
| **捕获** | 需要切换应用，打断心流 | 高亮/剪藏零切换，在阅读的地方记录 |
| **上下文** | 记录时已脱离原始语境 | 笔记和网页并排显示 |
| **重遇** | 需要用户主动打开工具回顾 | 浏览网页时自动触发 |
| **连接** | 靠用户手动链接或搜索 | 浏览器知道你正在看什么 |

### 与同类产品的关系

soma 的方法论吸收了四款产品的核心理念，但通过浏览器原生位置实现了它们做不到的闭环：

- **flomo 的"低摩擦记录"** → soma 的捕获比 flomo 更自然（高亮即记录，不需要切换到微信）
- **Tana 的"渐进式结构化"** → soma 用 Supertag + 字段实现同样的能力（先记后结构）
- **Roam 的"意外发现"** → 上下文感知侧边栏就是浏览器版的反向链接面板
- **Heptabase 的"理解优先"** → 大纲中的组织操作就是另一种形式的空间思维

### 三个场景，各司其职

```
浏览器侧边栏  — 最好的捕获和消化场景（读+记+整理）
IM 渠道       — 最好的触达和快速输入场景（推送+问答+速记）
大纲编辑器    — 最好的深度思考场景（组织=理解）
```

三个场景共享同一份知识，用户在任何一个入口都能推动正反馈循环。

## 五、设计原则

### 关于记录

1. **好的工具帮助用户更加专注，坏的工具分散注意力** — 每个功能都要问：这是帮用户专注，还是在抢夺注意力？
2. **少即是多，不是越少越好，而是增加信息密度** — 界面上的每个元素都应该传递有用信息，减少装饰和噪音
3. **工具即过程** — 通过工具的使用流程，帮助用户塑造思考的习惯，而不只是提供功能

### 关于消化

4. **用自己的话写，才算真正理解** — 费曼学习法的实践：笔记是把知识讲给未来的自己听
5. **标签是索引，不是分类** — 标签帮助未来找到笔记，而不是建立完美的知识体系。过度打标签不如没有标签
6. **删除是日常动作** — 如果真心舍不得某条内容，你会有写点东西、把核心价值保留下来的冲动。认同和触动不等于有用

### 关于重遇

7. **教育用户最好的时机是帮助他们实现正在积极努力的目标** — 笔记出现的最佳时机是用户正在处理相关问题的时候
8. **AI 帮你找到和使用旧笔记，而不是替你思考** — AI 是检索和连接的引擎，不是内容生成的替代品。"让 AI 替你思考"违背了记笔记的初衷

### 关于 AI

AI 在 soma 中有四个层次，价值逐层递增：

| 层次 | AI 做的事 | 用户感受 |
|------|----------|---------|
| **检索** | 找到相关的旧笔记 | "还好没丢" |
| **总结** | 压缩和格式化外部信息 | "工具很强" |
| **连接** | 在笔记之间发现关联 | "原来这两件事有关系" |
| **洞察** | 从用户的笔记中发现用户自己没意识到的模式 | **"原来我是这么想的"** |

前两层是工具能力，后两层才是认知增强。soma 的 AI 应该朝着第四层发力——不生成新内容，而是**把用户散落的想法照亮**：找到隐藏的矛盾、未被表述的共识、思维的转折点。

这是真正的"建构"：用户的知识没有增加，但用户对自己知识的**理解**加深了。 AI 不替你思考，而是把你自己的思考照亮。

### 关于产品

9. **正反馈是成功的关键，负反馈带来的更多是消耗** — 把记录从"任务"变成"收获"，关注"我今天又理解了 Y"而不是"我还没学会 X"
10. **接受不完美，Demo 先行** — 不要对着空泛的文本讨论，先做出来，基于实物迭代。迭代思维，不要总是重新开始
11. **如果工具所解决的问题并不存在，那么这个工具就不可能是好工具** — 每个功能都要回答：用户在没有这个功能时，真的痛吗？

## 六、soma 不做什么

| 不做 | 原因 |
|------|------|
| **不做 All-in-One** | 功能堆砌分散注意力，违背"少即是多" |
| **不做自动整理** | AI 替你整理 = 你没有消化，笔记依然是别人的知识 |
| **不做知识图谱可视化** | 看起来强大但使用门槛高，大多数人用不上 |
| **不做社交/分享** | 笔记是写给自己的，为别人写会扭曲记录的动机 |
| **不追求笔记数量** | 100 条消化过的笔记，胜过 10000 条未处理的剪藏 |

## 七、一句话总结

> **Think where you read.**

soma 让你在浏览的时候记下来，在整理的时候理解，在需要的时候用上。这是一个浏览器原生的知识循环：捕获靠低摩擦，消化靠大纲组织，重遇靠上下文匹配。三步闭环，正反馈驱动。

---

*"真正重要的不是未来会改变什么，而是什么永远不会改变。" — 贝佐斯*

*人们永远需要记录想法，永远需要在需要的时候找到它们。soma 围绕这两件不会改变的事来构建。*
