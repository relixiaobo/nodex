> - **Taste 派生规则存为 #skill 子节点**：原始修正信号来自 OpLog（零额外采集），Schema evolution skill 用 LLM 分析修正模式并派生规则，存为 #skill 子节点（可审查可修正）
> - **"安静"定义修正**：从"AI 不出现"改为"安静融入，非必要时静默，等待召唤"
> - **OpLog 分析需要 LLM**：修正"零 LLM token"的不准确表述——原始信号零额外采集（OpLog 天然存在），但修正模式分析需要 LLM
> - **不做 embedding**：删除 Related 面板、L1 Embedding 层、所有 embedding 相关内容。模型调用策略简化为 L0 本地规则 + L1 LLM 两级
> - **Chat 独立持久化**：Chat 对话不作为节点存储在图谱中，独立于 outliner 持久化。避免对话消息混入知识节点
>
> **2026-03-11 认知工作流修订**：
> - **三条认知工作流**：新增核心框架——阅读环（AI 处理结构 → 用户专注灵感）、碰撞环（Chat 加密想法密度）、回顾环（Review 照亮认知演化）。三条环闭合的是用户，不是软件
> - **价值叙事修正**：从"工具越用越懂你"转向"用户越用越清醒"。Schema evolution skill 是基础设施，不是核心飞轮
> - **Review 深化**：从内容摘要升级为认知镜像——发现新结构、升级、矛盾、同构。Review 对不同信息做**价值分层**——根据标签（#spark/#note）、连接密度、用户互动信号分配注意力权重，引导用户关注最值得花时间的内容
> - **Chat 碰撞模式**：Chat 不仅是执行通道，也是思维碰撞空间——用户带初生想法进入，AI 帮助加密度
> - **阅读工作流**：新增 AI 如何改变阅读本身的设计——从"提取信息"到"触发思考"
> - **Clip & Spark**：clip 从"剪藏"重新定位为"认知触发"——AI 提取结构（不是摘要）+ 与已有笔记碰撞（不是存储），处理逻辑是进化的 #skill（不是固定模板），价值随笔记密度增长（碰撞飞轮）。命名确定为 **Clip & Spark**：Clip = 采集动作（熟悉），Spark = 激发体验（独有）——火花是碰撞的结果，是起点不是终点
> - **Clip & Spark 节点结构**：#source 是顶层锚点（含 #article/#video/#social 子类型），#spark（AI 认知触发结果）和 #note（用户笔记）是 #source 的平级子节点，#highlight 作为 #source 的 Highlights 字段（默认折叠）。Spark 内容持久化为节点树，highlight 通过 reference 嵌入 spark/note 中作为论据。AI 生成的和用户写的自然共存，不做视觉区分
> - **Spark 触发策略**：用户主动 clip → 自动 Spark；用户先在网页 highlight/note、被动创建 clip → 手动 Spark 按钮；无 API key → Spark 按钮灰显但不阻塞其他功能
> - **Subagent 异步后台执行**：subagent 从"同步用完即弃"改为"异步后台 worker"——主 agent 委派复杂任务后立即回到 Chat 继续对话，不阻塞用户。Chat 面板右上角显示后台任务指示器（badge + 任务列表），用户可查看进度、取消任务。多个 subagent 可并发执行
> - **多 Agent 通信架构**：调研 10+ 框架（Google ADK / OpenAI Agents / LangGraph / Vercel AI SDK 等）后确定自建方案——在 pi-agent-core 上建 AgentOrchestrator（~200-300 行 TS，零新依赖）。双层通信：Loro CRDT 作为 blackboard（数据交换，一切皆节点）+ EventTarget 消息总线（协调信号：任务生命周期/求助/取消）。subagent 可中途求助主 agent（clarification flow）、subagent 间可通过 CRDT 协作。详见 `docs/research/multi-agent-orchestration.md`
>
> **2026-03-11 v5 内部审查修订**：
> - **Taste 学习改为基于 CRDT OpLog**：不再建独立的行为日志节点，Schema evolution skill 读 Loro OpLog + LLM 分析修正模式
> - **Review 与 taste 学习分离**：taste 学习 = 系统层静默优化（AI 怎么工作）；Review = 用户发起的内容反思（`/review`，照亮思考脉络），完全可选
> - **市场定位图修正**：soma 从右上角移到中上偏左（高结构精度 + 中等 AI 自主程度），加入"随 taste 成熟右移"的演进方向
> - **知识组织无客观正确答案**：作为底层哲学写入 §2，AI 的不同视角本身有价值
> - **增加目标用户画像**：想要知识被组织但怕麻烦的普通人，不同深度用户的使用方式不同但核心画像一致
>
> **2026-03-11 冷启动与商业模式决策**：
> - **检索策略确认：图结构 + 渐进式披露，不做 embedding**：碰撞和 Review 的语义检索依靠节点树结构（tags/fields/tree hierarchy）+ AI agent 渐进式发现（`node.search` + `node.read`），不引入向量嵌入。RAG 准确率不够，agent 驱动的结构化检索更适合找跨域同构
> - **碰撞是 pull 不是 push**：Spark 自动触发时只做结构提取 + 留白，碰撞仅在置信度高时出现（宁可不碰，不硬碰）。碰撞的主要入口是用户主动发起的 `/review` 和 Chat 碰撞模式——用户有预期时薄连接也不反感
> - **冷启动策略**：Day 1 差异化靠高质量结构提取（取景框，不是摘要）。Chat 碰撞模式从 Day 1 可用（通用知识加密度，不依赖笔记积累）。`/review today` 从 Day 1 可用（今天记录的节点之间找连接）。碰撞飞轮渐进增强，不人为制造
> - **商业模式方向**：先 BYOK（用户提供 API key）或 OAuth 授权（类似 Codex），soma 不直接提供 LLM。Outliner 本身收费（类似 Workflowy 模式）。后续根据情况开放订阅
> - **Reference 原则**：AI 提到任何已有节点时用 reference 而非纯文字，发现关联时物化为 reference。画板上 = 真实图谱边（inline ref / reference node），Chat 中 = 可点击导航链接（inline ref / citation 角标）。Reference 强化图结构检索、加速碰撞冷启动、让 AI 回答可验证
> - **Spark 三轮认知压缩**：参考李继刚「深度拆书机」+ Epiplexity 原理。Round 1 骨架（取景框）= #spark 顶层子节点，Round 2 血肉（论证链/隐形假设/边界条件）= 骨架的 children（展开查看），Round 3 灵魂（跨域映射/知识连接）= 碰撞环（Chat/Review，积累后渐进出现）。outliner 树状结构天然支持渐进式披露，不需要特殊 UI

## 一、定位

> **soma 是一个安静的知识环境，也是一面认知的镜子。打开就是今天，想到什么写什么，看到什么 clip 一下。Outliner 是用户和 AI 共同持有的画板——用户在上面自由记录，AI 在用户需要时通过 chat 指令在同一块画板上行动。三条认知工作流——阅读时捕捉灵感、Chat 中碰撞想法、Review 时照亮认知演化——让用户在使用中越看越清楚。AI 的静默学习是让这个循环越转越顺的基础设施。**

### 为什么独特

1. **零概念启动** — 没有 Library/Inbox/Notebook，打开就是今天，写就行
2. **输入即保留** — 用人话写，原样保存，不被 AI 篡改或打断
3. **Chat 双模式** — 执行模式（帮我整理）+ 碰撞模式（我刚想到一个东西），AI 既执行也碰撞
4. **共享画板** — outliner 是用户和 AI 的共享工作面，谁画的都可见可修正
5. **Schema 由 AI 维护** — 用户不需要理解 supertag/field/data type
6. **三条认知环** — 阅读时捕捉灵感、Chat 中碰撞想法、Review 时照亮认知演化——用户越用越清醒
7. **静默学习** — AI 在幕后从修正模式中迭代规则，是让三条环越转越顺的基础设施
8. **浏览器原生** — 浏览器就是数据采集界面
9. **安静** — AI 安静融入工作流，非必要时完全静默，等待用户召唤

### 市场空白

```
        结构精度高
            ↑
   Tana ●   │              ← 结构强但需要用户先建
Notion ●    │
Capacities ●│  ☆ soma
            │  "用户召唤 + AI 执行 + 静默学习"
            │                     ● Recall
            │              ● Mem 2.0
            │         ● Napkin
            │    ● Fabric
            │ ● mymind
────────────┼────────────────────→ AI 自主程度高
            │ ● Sider/Monica
            ↓
        结构精度低
```

soma 的位置：**高结构精度 + 中等 AI 自主程度**。与 Tana 同级的数据模型，但 AI 只在用户召唤时行动（chat/clip/tag），加上基于 CRDT OpLog 的静默 taste 学习让 AI 逐渐变准。核心价值不是"AI 越来越懂你"（那是基础设施），而是**三条认知工作流让用户越用越清醒**。

与右上角（高结构 + 高自主）的区别：soma 不替用户做组织决策，而是在用户召唤时高质量执行。随着 taste 学习成熟，AI 行动的准确度提升，soma 会**沿箭头方向自然右移**——但始终是用户主动触发，不是 AI 自作主张。

### 目标用户

**一句话**：想要自己的知识被组织，但又怕麻烦的普通人。

这是一个知识管理领域长期存在的需求——从 Evernote 到 Notion 到 Tana，不断有产品试图解决"知识组织"问题。用户画像始终是同一群人：有记录和整理信息的需求，但不愿意花大量时间学习工具概念或手动维护结构。

soma 的立场是：**通过 AI 降低组织摩擦，通过三条认知工作流帮用户看清自己的思考**。AI 处理结构（干），用户专注于灵感、碰撞和反思（湿）。工具优化是基础设施，认知演化才是核心价值。

不同类型用户的使用深度不同，但**核心画像不变**：

| 用户类型 | 怎么用 soma | soma 对他们的价值 |
|---------|------------|------------------|
| **轻度用户** | 随手记、clip、偶尔 ⌘K 让 AI 整理 | 零配置的结构化笔记，AI 帮忙整理 |
| **中度用户** | 自建节点组织结构、创建 tag、用 chat 做复杂任务 | outliner + AI 协作的知识工作台 |
| **深度用户** | 手动精细组织 + /review 反思 + 微调 schema | outliner 本身就是强大的组织工具，AI 是加速器 |

**关键设计约束**：轻度用户永远不应该需要理解 schema、field type、skill 这些概念。他们看到的只是"写了 → AI 帮我整理了 → 越来越顺手"。

## 二、五个核心张力

所有研究最终指向五个互相纠缠的张力。定位的本质是选择如何解决这些张力。

### 张力 1：结构 vs 自由

**Tana/Notion 路线**：用户先建结构（supertag、field、database）→ AI 在结构内运作。
**Fabric/Mem 路线**：用户投入内容 → AI 自动组织。

110a 的市场数据表明**两个极端都失败了**：
- Fabric 用户："我的东西去哪了？"——失去心智模型
- Tana 用户：维护结构的时间 > 使用系统的时间——AI 时代这更不可接受

**未被占领的中间地带**：结构是 AI + 用户行为的**共同产物**，不是纯用户输入也不是纯 AI 输出。

### 张力 2：Schema = Prompt vs 用户不想维护 Schema

107 的核心洞察是"supertag schema 就是 AI 的提取模板"——用户定义 #article 的 Author 字段，等于告诉 AI 要提取作者。

但对话中的认识是：**大多数用户不会主动定义 schema**。他们愿意做"选择"（clip、高亮、标记重要），不愿意做"定义"（配置字段名、选数据类型、排列顺序）。

这两个洞察看似矛盾，实际上指向同一个方向：**Schema 仍然是 AI 的提取模板，但 schema 本身也应该由 AI 从用户的行为中涌现出来。**

### 张力 3：Think Now vs 无负担输入

产品方法论说："不经过思考的信息，未来大概率用不上。"——核心信条是"用自己的话写，才算真正理解"。

codex 110 说："让用户无负担输入。"——原文可以先进入系统作为 Source，理解和提炼可以延迟。

这两个观点的冲突比看上去的小。关键区分是：**用户选择 clip 这个动作本身就是一种思考**——它表达了"这对我有意义"的判断。它不是无脑囤积，它是经过 taste 筛选的选择。

真正的问题不是"是否允许原文进入"，而是**原文进入后的身份是什么**：
- 如果原文 = 知识 → 鼓励囤积，违背方法论
- 如果原文 = Source（待提炼的材料）→ 用户的选择被尊重，但系统仍然期待提炼

### 张力 4：Outliner vs AI 交互

109 的结论：outliner 是结构工具（组织模式），AI chat 是探索工具（对话模式），两者认知模式不同，强行统一让两边都别扭。

更深的问题是：**如果 soma 的核心界面就是 outliner，那 AI 住在哪里？** Side panel 只有 300-700px，放不下两个界面。

v2-v4 的新认识经过了一个演变：先是"outliner 是共享工作面，AI 在上面自动行动"→ 交互分析后发现这会让 outliner 变慢变吵。

**v5 的解法：输入通道分离，输出画板共享。** Outliner 是用户和 AI 共同持有的画板——用户直接在上面画（node 输入，原样保留），用户也可以让 AI 在上面画（通过 chat 指令）。两者的作品在画板上形式完全一样（都是节点），但发起方式不同。Chat 是 AI 的指令通道，outliner 是两者共享的结果面。

### 张力 5：产品方法论的"不做自动整理" vs AI 辅助结构涌现

方法论明确说："AI 替你整理 = 你跳过了 Think 和 Connect，笔记依然是别人的知识。"

但我们现在认为"AI 应该帮助结构从用户行为中涌现"——这看起来就是自动整理。

关键区别不在于"谁做决定"（提议→确认仍然是审批流程），而在于**可见性和可修正性**：
- 不透明的自动整理 = AI 决定结构，用户看不到也改不了 → 跳过 Connect，AI 永远不懂用户
- 透明的 AI 协作 = AI 在可见的节点树上行动，用户随时可以修正 → 修正本身就是 Connect，且教会了 AI 用户的 taste

### 底层认识：知识组织没有客观正确答案

五个张力的底层有一个共同的哲学前提，值得显式写出来：

**知识组织不像代码编译——没有"通过/失败"的客观标准。** 不同的取景框看到不同的风景。一篇关于 AI 产品设计的文章，标 #AI 和标 #Product Design 都有道理——这不是"对/错"的问题，而是"视角"的问题。

这个认识有几个重要推论：

1. **AI 的"错误"分类可能恰好是惊喜** — 用户没想到的视角可能带来新连接，好过千篇一律的分类。这是 AI 参与知识组织的独特魅力
2. **"正确率"不是最重要的指标** — 真正有害的不是"不完全准确"，而是"毫无道理"。只要 AI 的组织有内在逻辑，它就是有价值的起点
3. **用户对 AI 结果的三种反应都是正向的** — 没注意到（不在乎）、看到没改（默认接受）、看到改了（taste 信号）。不存在"AI 做了但用户永远不知道因此受损"的场景——最多是"无感"
4. **这让 soma 比其他 AI 领域更适合激进信任** — 邮件分错可能错过重要信息（真实成本），代码改错破坏构建（客观错误）。知识组织的"错误"成本极低，且可能是特性而非 bug

## 三、认识的演进（v1 → v5）

### v1 认识（105-108）

> Schema = Prompt：用户定义结构 → AI 执行。用户的组织行为就是在配置 AI。

**问题**：假设用户愿意且能够定义 schema。

### v2 认识（109 + 对话）

> 用户不想维护结构。维护结构本身就很不 AI。Schema 应该由 AI 从选择中涌现，通过提议→确认流程沉淀。

**问题**：提议→确认仍然是一种审批流程，不像协作，不像导师。

### v3 认识（112 + 权限模型讨论）

> **AI 直接行动，不问。结构通过 AI 的理解自然体现在信息的呈现中。用户通过正常使用自然修正。修正是最高价值的 taste 信号——它告诉 AI 用户的思维方式。**

核心转变：
- 从"用户确认 AI 的提议" → **"AI 行动，用户使用中自然修正"**
- 从"确认是协作" → **"修正是协作"**
- 从"结构需要被审批" → **"结构在使用中被验证"**

### v4 认识（113 + 信息架构简化）

> **去掉一切强加的概念。日期节点是唯一锚点，其他结构全部涌现。Schema 由 AI 维护，用户只需输入。soma 是一个安静的知识环境——用得越久，越懂你。**

核心转变：
- 从"Library + Inbox + Journal" → **"只有日期节点，其他用户自己建"**
- 从"Schema 是用户的配置工具" → **"Schema 是 AI 的工作台，用户可审查"**
- 从"AI 是一个功能" → **"AI 融入环境，用户感觉不到 AI 的存在"**
- 从"知识协作者" → **"安静的知识环境"**——最好的 AI 体验是用户感觉不到 AI

**问题**：交互分析发现"AI 在每次 Enter 时结构化"会让 outliner 变慢变吵——ghost UI 到处冒、布局跳动、打断思维流。real-time taste learning 信号稀疏且归因困难，飞轮转不起来。

### v5 认识（交互可行性分析 + 输入分离）

> **Outliner 是用户和 AI 共同持有的画板。用户直接在画板上记录（原样保留），需要 AI 时通过 chat 指令让 AI 在画板上行动。AI 在后台从使用模式中静默学习，用户可随时 Review 照亮自己的思考脉络。**

核心转变：
- 从"AI 在每次 Enter 时结构化" → **"Node 输入原样保留，AI 只在 chat/clip/tag 创建时介入"**
- 从"AI 无处不在但你感觉不到" → **"AI 安静融入工作流，非必要时静默，等待召唤"**
- 从"修正驱动飞轮（实时学习）" → **"基于 CRDT OpLog 的静默 taste 学习（系统层）+ 用户发起 Review（内容层反思）"**
- 从"Ghost UI 随处浮现" → **"Ghost UI 只在 tag/field 创建时出现"**

**问题**：价值叙事仍停留在"工具越用越懂你"——把 AI 变聪明当核心卖点。v6 架构简化解决了 agent 模型的复杂度，但没有回答"用户为什么越用越离不开"。

### v6 认识（当前，认知工作流 + 架构简化）

> **soma 是一面认知的镜子。用户越用越清醒——不是因为工具变聪明了，而是因为三条认知工作流（阅读环、碰撞环、回顾环）让用户在使用中看清自己的思考。AI 的静默学习是让这个循环越转越顺的基础设施，不是价值本身。**

核心转变：
- 从"工具越用越懂你"（v5 价值叙事）→ **"用户越用越清醒"——环闭合的是用户的认知，不是软件的模型**
- 从"taste 学习 = 核心飞轮" → **"三条认知环 = 核心价值，taste 学习 = 基础设施（润滑剂）"**
- 从"Review = 内容摘要工具" → **"Review = 认知镜像——照亮新结构、升级、矛盾、同构"**
- 从"Chat = 执行通道" → **"Chat = 执行模式 + 碰撞模式（思维加密器）"**
- 从"五层 agent 架构" → **"节点 + Tools——agent 的一切（身份、规则、技能、记忆）= 节点"**

## 四、市场空白

综合 105 和 110a 的全景分析，市场上 20+ 产品的分布如下：

```
        结构精度高
            ↑
            │
   Tana ●   │              ← 结构强但需要用户先建
Notion ●    │
Capacities ●│  ☆ soma
            │  "用户召唤 + AI 执行 + 静默学习"
            │                 ───→ taste 成熟后自然右移
            │
            │                     ● Recall
            │              ● Mem 2.0
            │         ● Napkin
            │    ● Fabric
            │ ● mymind
────────────┼────────────────────→ AI 自主程度高
    用户     │              AI
    控制结构  │            控制结构
            │
            │ ● Sider/Monica（无结构）
            │
            ↓
        结构精度低
```

**soma 的位置**：高结构精度 + 中等 AI 自主程度。不在右上角——v5 的 AI 只在用户召唤时行动，不自作主张。

**与右侧产品的区别**：Recall/Mem/Fabric 的 AI 自主程度更高（全自动），但结构精度低（不可见/不精细/不可修正）。soma 选择**先保证结构精度，再通过 taste 学习逐步提升 AI 准确度**。随着 taste 成熟，AI 行动越来越准，用户越来越信任，soma 自然右移——但始终是用户掌控节奏。

**当前状态 vs 方向**：
1. **结构精度是已实现的** ✓（Tana 级数据模型 + outliner 天然可视可修正）
2. **AI 能力是方向性的** ○（chat/clip 触发 + taste 学习 + Schema 涌现——这是要建的，不是已有的）
3. **为什么这个方向可信**：soma 已有 #1（数据模型）和 #3（outliner 可视可修正），这两个是最难从零构建的基础设施。#2（AI 涌现结构）是在已有基础上叠加能力层。Fabric/Mem 反过来——有 #2 但缺 #1 和 #3，补结构化数据模型比补 AI 能力难得多

## 五、定位假说

### 一句话

> **soma 是一面认知的镜子。你用得越久，看自己越清楚。**

核心价值不是"工具变聪明"——而是用户在阅读、碰撞、回顾的循环中，看清自己的思考脉络。AI 在幕后的静默学习让这个循环越转越顺，但那是基础设施，不是价值本身。

### 三条认知工作流

soma 的核心价值不是"工具变聪明"，而是**用户在三条认知工作流的循环中，越看越清楚**。AI 在幕后的静默学习（Schema evolution skill）是让循环越转越顺的基础设施，不是价值本身。

**阅读环（Reading Ring）**

```
浏览网页
  ↓
AI 处理结构（干）：提取思维框架、与已有笔记碰撞
  ↓
用户被释放，专注于"这让我想到什么"（湿）
  ↓
捕捉的是自己的想法，不是原文的结构
  ↓
想法积累改变了用户的取景框
  ↓
同一篇文章，三个月前的你和三个月后的你读出完全不同的东西
  ↓
环闭合：不是软件更懂你，是你更懂你自己
```

阅读的目的从"从文章中提取信息"变成了"让阅读触发自己的思考"。AI 接管了结构提取（干），用户被释放出来做只有人能做的事——感受共鸣、捕捉灵感（湿）。soma 侧边栏不是更好的剪藏工具，而是**认知触发器**——AI 提取内容的思维结构、与已有笔记碰撞，用户看到碰撞结果后写下自己的想法（涟漪）。原文只是触发器，早已退场。

**碰撞环（Collision Ring）**

```
阅读触发了一个初生的想法（湿）
  ↓
打开 Chat，不是"帮我做什么"，而是"我刚想到一个东西"
  ↓
AI 带入多角度、拉高密度（干）
  ↓
碰撞中有些方向用户觉得"这个有意思"（湿的判断）
  ↓
产生的不是结论，是新的问题、实验、观察方向
  ↓
这些方向改变了用户下次阅读时的取景框
  ↓
环闭合：用户的认知结构在生长
```

Chat 面板的深层定位不只是"执行通道"（帮我整理笔记），还是**思维的加密器**。用户带着一个模糊的直觉进来，通过对话让它变得致密，然后带着更清晰的问题离开。输入到碰撞中的不是原文，而是用户自己的想法——原文只是触发器，早就退场了。

**回顾环（Review Ring）**

```
用户持续使用 soma（clip、写笔记、碰撞、整理）
  ↓
内容积累到一定密度
  ↓
/review → AI 分析的不是"你存了什么"，而是"你的认知发生了什么"
  ↓
  · 这段时间你的认知结构上哪些新结构出现了
  · 哪个旧结构被升级了
  · 哪里出现了矛盾/张力你还没解决
  · 两篇看似无关的笔记底下藏着同构
  ↓
用户看见自己的认知轮廓 → 产生新的反思和想法
  ↓
反思本身又成为新的节点 → 环闭合
```

Review 不是内容摘要（"这周你写了 8 条笔记"），而是**认知镜像**——让用户看见自己看不见的东西。你以为你这个月在研究 A，但 Review 告诉你其实你真正在意的是 B，而且 B 和你三个月前放弃的 C 之间有一个你没注意到的同构。

**三条工作流的共同结构**

| 工作流 | 什么在生长 | AI 做什么（干） | 用户做什么（湿） |
|---|---|---|---|
| 阅读环 | 用户的取景框 | 结构提取，释放用户注意力 | 感受共鸣，捕捉灵感 |
| 碰撞环 | 用户的想法密度 | 多角度碰撞，找同构/冲突 | 判断哪个方向值得追 |
| 回顾环 | 用户的自我认知 | 分析认知结构变化 | 看见自己，反思调整 |

三条环合在一起是一个人的认知飞轮：**读 → 想 → 看见自己 → 带着新的眼睛去读**。每条环闭合的都不是软件——是用户自己。

### 用户的全部动作

**阅读 → 记录 → 碰撞 → 回顾。**

| 动作 | 做什么 | 对应的认知环 |
|------|-------|-------------|
| **阅读** | 浏览网页，AI 预处理结构，用户专注于灵感捕捉 | 阅读环 |
| **记录（Node 输入）** | 随手写自己的想法、拖拽整理、Clip 触发后写下涟漪 | 阅读环（输出）|
| **碰撞（Chat）** | 带着初生想法进入 Chat，AI 加密度、多角度碰撞 | 碰撞环 |
| **回顾（Review）** | `/review this month` — AI 照亮认知结构变化 | 回顾环 |
| **指令（Chat 执行模式）** | "帮我整理笔记"、"提取这个网页" — AI 在画板上执行 | 基础设施 |

### 飞轮

```
阅读环：浏览 → AI 处理结构 → 用户捕捉灵感 → 取景框改变 → 读出不同的东西
    ↕
碰撞环：初生想法 → Chat 加密度 → 新的问题和方向 → 取景框改变
    ↕
回顾环：积累 → Review 照亮认知演化 → 看见自己 → 改变下次阅读和思考
    ↕
  ┌───────────────────────────────────┐
  │  基础设施：Schema evolution skill   │
  │  OpLog → 修正模式 → #skill 更新    │
  │  让三条环中 AI 的「干」越来越准      │
  └───────────────────────────────────┘
```

**认知飞轮（核心价值，用户可感知）**：三条环共同驱动用户的认知演化。阅读产生灵感，灵感在碰撞中致密化，回顾照亮积累中的模式——每条环的闭合都改变用户下次进入循环时的状态。

**基础设施飞轮（润滑剂，用户无感）**：Schema evolution skill 在后台从 OpLog 修正模式中迭代 #skill 节点规则。这让三条认知环中 AI 的「干」的部分越来越准——结构提取更精准、碰撞时上下文更丰富、Review 时模式识别更深入。但这是让环转得更顺的润滑剂，不是环本身。

**为什么比 v1-v5 更好**：
- v1（Schema = Prompt）要求用户先建 schema → 冷启动问题
- v2（提议→确认）需要显式审批步骤 → 打断流程
- v3（行动→修正）零额外步骤，但还有 Library/Inbox 等预设概念
- v4（自动结构化）AI 在每次 Enter 时行动 → outliner 变慢变吵，taste 信号稀疏
- v5（画板 + 双飞轮）输入不被打断，AI 只在被召唤时行动——但价值叙事仍停留在"工具变聪明"
- **v6（三条认知环 + 基础设施）**价值叙事回归用户自身的认知演化。工具优化是基础设施，用户看清自己才是目的

### 与竞品的差异

| | Tana | Fabric | Mem | soma |
|---|---|---|---|---|
| 启动成本 | 高（学概念+建结构） | 低 | 低 | **零（打开就是今天）** |
| 输入方式 | 选标签→填字段 | 自由文本 | 自由文本 | **自由文本（原样保留）+ Chat 指令** |
| AI 介入时机 | 有限（内置命令） | 全自动 | 全自动 | **用户召唤时（chat/clip）** |
| Schema 维护 | 用户手动 | 无 schema | 无 schema | **AI 维护，用户可审查** |
| 结构来源 | 用户预建 | AI 全自动 | AI 全自动 | **用户 + AI 共同画板** |
| 结构可见性 | 完全可见 | 不透明 | 部分可见 | 完全可见（outliner） |
| AI 学习方式 | 无 | 静默 | 静默 | **静默 taste 学习（基础设施）+ 三条认知环（核心价值）** |
| 预设概念 | 多（Workspace/Space/Schema/...) | 少 | 少 | **几乎零（日期节点+自建）** |
| 浏览器原生 | 否 | 否 | 否 | 是 |

## 六、Outliner 的重新定位：共享画板

### 旧理解（v1-v3）

Outliner 是一个让用户手动组织知识的工具。它的价值在树操作：缩进、拖拽、展开折叠。

### v4 理解

> Outliner 是 AI 和用户之间的共享工作面。AI 在上面自动行动，用户在上面修正。

**问题**：交互分析发现，AI 在每次 Enter 时自动行动（加 ghost tags、填字段）会让 outliner 变慢变吵——布局跳动、注意力打断、快速连续输入时尤其明显。

### v5+ 理解

> **Outliner 是用户和 AI 共同持有的画板。共享的是画板（输出面），不是画笔（输入方式）。**

```
┌─────────────────────────────────────────┐
│              Outliner（画板）              │
│                                         │
│  用户画的        Agent 画的               │
│  ├── 随手写      ├── clip 提取的结构       │
│  ├── 拖拽整理    ├── chat 指令的执行结果    │
│  └── 手动加标签  └── tag/field 建议被接受的 │
│                                         │
│  同一个画布，同一种笔触（节点），           │
│  谁画的可以区分，但形式完全一样             │
└─────────────────────────────────────────┘
```

区别在于**谁发起**，不在于**在哪里呈现**：

- 用户直接在画板上画（node 输入）→ 原样保留，AI 不碰
- 用户对 agent 说"你来画"（chat 指令）→ agent 在同一块画板上操作
- 用户 clip 了一个东西（认知触发）→ agent 提取结构、碰撞已有笔记，结果放上画板
- 用户在建 tag/field（定义结构）→ agent 给建议，用户决定是否采纳

**画板是两者唯一的共享事实来源。** Chat 是临时的指令通道，画板是持久的知识载体。agent 所有的"思考"最终都要表达为画板上的节点操作——移动、标签、连接、创建。

### 为什么画板模型比"自动行动"更好

**这也解释了为什么"一切皆节点"在 AI 时代的价值**：不是让 AI 自动在节点上加标签，而是让节点树成为 AI 和用户协作的**唯一事实来源**。AI 通过 chat 接收指令，通过画板呈现结果，用户在同一个画板上审查和修正。双方在同一个数据模型上工作，但输入通道各自独立。

**Fabric 和 Mem 的问题不是"AI 自动做"，而是"做了之后看不到"**——它们没有一个透明的画板。soma 的 outliner 天然是这个画板。

### Outliner vs Chat 的解决

109 的问题是"outliner 不适合 AI 对话"。在 v5 画板模型下，两者各司其职：

- **Outliner（画板）= 共享工作面**——用户和 AI 的所有工作结果都呈现在这里
- **Chat（面板）= 对话面**——用户与 AI 的对话界面，独立面板，历史持久化
- **两者职责清晰，不需要合一**——Chat 是对话和协作的地方，画板是结果的呈现
- **执行→报告→反馈**——AI 直接在画板上操作，在 Chat 中报告做了什么，用户可以同意 / 给建议优化 / 撤回

## 七、极简信息架构：日期节点 + 涌现结构

### 旧架构：强加的容器

之前 soma 预定义了 Library、Inbox、Journal、Schema 等系统容器。Library 和 Inbox 是我们替用户做的组织决策——"你的东西应该分成收件箱和图书馆"。这与"结构从行为中涌现"的原则矛盾。

### 新架构：只保留不可替代的

| 保留 | 原因 | 实现方式 |
|------|------|---------|
| **Day nodes（日期节点）** | 时间是客观的锚点，不是强加的结构 | 系统自动创建 |
| **Trash** | 删除机制，不是组织概念 | 普通 node + locked |
| **Schema** | 元结构（定义结构的结构），定位变了——见下文 | 普通 node + locked |

**删除**：Library、Inbox。如果用户需要，自己建一个节点作为父节点即可。

### 消除 Container Node 概念

**旧模型**：`WORKSPACE_CONTAINERS` 定义 6 种特殊容器节点（Library、Inbox、Journal、Trash、Schema、Settings），ID 格式为 `{workspaceId}_{SUFFIX}`，是独立于普通 node 的特殊类型。

**新模型**：**Container node 不存在。** 普通 node + `locked` 属性 = 之前的 container。

Container 的所有"特殊性"都可以用已有机制实现：
- 不能删除 → `locked`
- 不能编辑名称 → `locked`
- 顶层导航锚点 → 根级节点（无 parent）
- 特定 UI 渲染 → 通过 supertag 标记（如 `#system-trash`）

**"一切皆节点"又纯粹了一步**——数据模型里只有 node 和 tuple，没有第三种东西。

### AI 结构化 = 加层，不是替换

用户在 outliner 中写什么，原样保留。结构化只在用户**主动召唤**时发生（chat 指令 / clip）：

```
用户在 outliner 中写: "刚和老王聊了，他觉得搜索优先级比AI高"
  → 原样保存，AI 不碰

之后用户按 ⌘K: "帮我整理一下今天的笔记"
  → 主 agent 分析，在画板上加层：

节点 name（保留原文）: 刚和老王聊了，他觉得搜索优先级比AI高
  #meeting                    ← 主 agent 加的标签
  People: 老王 →              ← 主 agent 加的字段
  Decision: 搜索优先级 > AI    ← 主 agent 加的字段
```

原文永远是节点的 `name`，结构化信息是标签和字段（Tuple）。AI 做错了，删掉标签/字段就行，原文不受影响。**关键区别：结构化是用户召唤的结果，不是每次 Enter 的自动动作。**

### Schema 的重新定位

**旧定位**：用户配置结构的地方——需要用户理解 supertag、field、data type 等概念。

**新定位**：**AI 的工作台，用户的审查视图。**

- Schema 主要由 AI 在维护——从用户的输入和行为中发现模式，创建和演化 supertag/field
- 用户**只需要不停输入**——AI 在幕后处理结构化
- 遇到 AI 维护得不对的，才需要进 Schema 页面调整
- **大多数用户永远不会打开 Schema 页面**。Power user 偶尔进去微调

类比 Git：你平时不需要看 `.git` 目录，但想看的时候，所有版本历史都在那里，完全可审查、可修改。

### 用户的心智模型

```
打开 soma → 就是今天 → 想到什么写什么 → 看到什么 clip 一下看看 AI 怎么说 → 写下自己的想法
```

零概念需要学习。没有"先了解 Library 和 Inbox 的区别"，没有"请先创建你的第一个标签"。

### Schema 涌现的四个阶段

Schema 不是用户一次性定义的，而是通过四个阶段从行为中逐步涌现：

**阶段 1：行为观察**
```
用户输入                          AI 记录
─────────────────────────────────────────
输入 "和老王聊了搜索优先级"       → 实体: 老王(人), 搜索(概念)
输入 "和小李讨论了部署方案"       → 实体: 小李(人), 部署(概念)
输入 "和老王回顾了Q1目标"        → 实体: 老王(人), Q1(时间)
```

**阶段 2：模式发现**
```
AI 发现模式: 用户频繁记录"和[人]聊了[主题]"
→ 推断潜在标签: #meeting
→ 推断潜在字段: 参与者(person)、主题(text)
```

**阶段 3：Schema 应用（通过 chat/clip 触发）**

AI 用发现的模式结构化内容——但只在用户主动召唤时（chat 指令 / clip 动作）。例如用户 clip 了一篇文章，AI 用已有 schema 提取字段；用户按 ⌘K "整理今天的笔记"，AI 用已有 schema 加标签。Schema 本身的创建和演化通过 Schema evolution skill（扫描 CRDT OpLog + LLM 分析修正模式）静默发现 + 在 chat/clip 中自然应用。

**阶段 4：Schema 演化（静默学习 + 自然应用）**

Schema 演化通过两条路径发生：
- **静默路径**：Schema evolution skill 从 OpLog 分析模式（如 80% 的 #meeting 提到结论）→ 更新 #skill 节点规则 → AI 下次 chat 整理时自动添加 Decision 字段 → 用户在画板上看到，不对就删
- **显式路径**：用户在 tag/field 创建时，AI 基于已有模式给出 Ghost 建议

关键：阶段 1-2 完全不可见（Layer 0）；阶段 3 在 chat/clip 通道内执行，画板上可见可修正；阶段 4 通过静默学习 + 自然应用演化，无需独立的审批环节。

### 结构如何涌现

- **Day 1**：所有内容挂在日期节点下。零容器。
- **Week 2**：用户按 ⌘K "帮我整理一下"，AI 发现 15 条内容中有几条相关的，建议归到一起
- **Month 1**：用户的顶层结构完全是自己长出来的——有人有 3 个顶层节点，有人有 10 个，每个人不一样
- **Month 3**：Schema 页面里有 AI 维护的 5-10 个 supertag，每个都是从行为中涌现的

**日期节点是唯一的系统锚点，其他结构全部涌现。**

### 与 AI 交互的配合

113 研究的"自然语言 → 结构化"在 v5 架构下通过 chat/clip 通道自然实现：

```
用户在 outliner 写: "刚和老王聊了搜索优先级"
  → 原样保存在今天的日期节点下

用户按 ⌘K: "帮我整理一下今天的笔记"
  → 主 agent 理解 + 结构化（标签、字段、引用）
  → 结果在画板上可见可修正

后台: CRDT OpLog 记录所有操作（用户接受/修正了哪些标签和字段）
  → Schema evolution skill 分析修正模式 → 更新 #skill 节点，下次行动更准确
```

用户不需要知道 Schema 发生了什么。他看到的是：主 agent 在画板上把笔记整理好了，如果不对就改。

## 八、AI 权限模型：通道分级 + 危险拦截

### 112 研究的结论 vs soma 的选择

112 研究了 12+ 主流产品的 AI 协作模式，发现了一个信任光谱：

```
← 保守                                                    激进 →

[AI 只看不动]  [AI 创建平行视图]  [AI 建议+确认]  [AI 自动执行可撤]  [AI 直接修改]
  YouTube推荐    Apple回忆         Copilot          Gmail分类          ？
               Google相册分类     Grammarly         Superhuman
                                 Linear建议        Notion Autofill
```

大多数产品停留在中间偏左（建议+确认）。112 的结论是保守的："没有主流产品处于最右端"。

**但 soma 的选择是：直接走到光谱的右侧。**

### 为什么 soma 可以比其他产品更激进

**类比：Claude Code 的权限演进。**

Claude Code 的用户趋势：越来越多人不再想要永远确认，只要危险操作得到确认即可。从 `default`（每次确认）→ `auto`（自动执行，危险拦截）→ `dangerouslySkipPermissions`（全权委托）。用户愿意给予更高权限，因为**更高的权限带来更高的可能性**。

soma 天然具备比大多数产品更好的安全网——**Loro CRDT 保留完整操作历史**：

| 操作 | 可逆性 | 机制 |
|------|--------|------|
| 添加/修改节点 | 完全可逆 | `loroDoc.undo()` |
| 删除节点 | 完全可逆 | Trash + undo |
| 重组结构（移动、重排） | 完全可逆 | undo |
| 标签/字段变更 | 完全可逆 | undo |
| 批量操作 | 完全可逆 | undo（单次 commit = 单次 undo） |

对比其他领域：
- **邮件**（Gmail）：分错可能错过重要信息 → 有真实成本
- **代码**（Copilot）：改错可能破坏构建 → 有客观的对/错判断
- **知识管理**（soma）：**组织没有客观正确答案**

**知识组织与代码/邮件的本质区别**：代码有编译通过/失败的客观标准，但知识组织是主观的——不同的取景框看到不同的风景。AI 给一篇文章标 #AI 而用户觉得应该是 #Product Design，这不是"AI 错了"，而是"AI 提供了另一个视角"。这种意外的视角可能带来惊喜和新连接，好过千篇一律的分类。

用户对 AI 组织结果只有三种反应，每种都是正向的：
1. **没注意到** → 用户不在乎具体组织方式，主要通过 chat 使用知识。组织是手段不是目的
2. **看到了没改** → 说明觉得合理，或无关紧要。两者都是有效的 taste 信号（OpLog 中的"未修正"）
3. **看到了改了** → 修正信号进入 OpLog → Schema evolution skill 分析 → 更新 #skill 节点，下次更准

**结论：知识管理领域的"AI 犯错"成本远低于其他领域，且 AI 的不同视角本身有价值。** 加上 CRDT undo 作为安全网，soma 可以在信任光谱上比大多数产品更靠右。

### soma 的权限模型

v5 的输入分离改变了权限模型的适用范围——"默认执行"不再指"AI 在每次输入时自动行动"，而是指"AI 在被授权的通道内直接执行"：

```
┌─────────────────────────────────────────────────┐
│  Chat 指令 → 默认执行（用户说了，直接做）          │
│                                                   │
│  · "帮我整理今天的笔记" → 直接组织                 │
│  · "给这几条加上标签" → 直接添加                   │
│  · "从这个网页提取信息" → 直接提取并创建节点        │
│                                                   │
│  所有操作在 outliner 中可见 + CRDT undo 可逆       │
├─────────────────────────────────────────────────┤
│  Clip → 认知触发（提取结构 + 碰撞已有笔记 + is/has/about 元数据）│
│                                                   │
│  · is: 识别类型 → 应用已有 supertag（#article）    │
│  · has: 按模板提取字段 → 直接填充（Author, Source） │
│  · about: 如有 Topics 引用字段 → 填充主题引用      │
├─────────────────────────────────────────────────┤
│  Tag/Field 创建 → 建议模式（AI 给建议，用户决定）   │
│                                                   │
│  · 创建 #meeting → AI 建议字段（People, Topic...） │
│  · 添加 field → AI 建议类型和 options               │
├─────────────────────────────────────────────────┤
│  Node 输入 → AI 不介入                             │
│                                                   │
│  · 用户在 outliner 中写什么，保持原样               │
│  · 只有 @/# 触发已有的规则匹配（毫秒级，确定性）    │
├─────────────────────────────────────────────────┤
│  危险拦截（任何通道都需确认）                       │
│                                                   │
│  · 批量删除（超过 N 个节点）                       │
│  · 跨工作区操作（影响共享数据）                     │
│  · 不可逆的外部动作（发送、发布、分享）             │
└─────────────────────────────────────────────────┘
```

### 为什么这比 v4 的"全面默认执行"更好

v4 说"直接执行 + 可逆 = 真正的协作"，逻辑上成立但交互上有问题——AI 在用户不期待的时候行动（每次 Enter），即使可逆也会打断思维流。

v5 的关键区分：**AI 在用户主动召唤时默认执行（chat/clip），在用户自由记录时完全不介入（node 输入）。** 用户在乎的不是 AI 是否征求了他的同意，而是**AI 是否出现在了正确的时机**。Chat 就是正确的时机——用户主动发起，预期 AI 行动，延迟可接受。

### 渐进权限扩展

与 Claude Code 一样，权限可以随信任建立而扩展：

| 阶段 | AI 行为 | 用户感知 |
|------|---------|---------|
| 冷启动（前 10 条） | 极保守：只做高置信度标签 + 类型识别 | "还行，基本准确" |
| 学习期（10-100 条） | 中等：开始组织结构、填充字段 | "有些不对，改一改" |
| 成熟期（100+ 条） | 积极：基于 taste 模型全面行动 | "越来越顺手了" |
| 高信任（用户主动开放） | 可配置更高权限（自动归档、自动合并等） | "全权交给你" |

### 浏览器权限：debugger 是安静 AI 的基础设施

soma 的 AI 需要**在网页上操作**（提取动态内容、点击展开、翻页滚动）和**验证操作结果**（截图确认）。Chrome 提供三层能力：

| 能力 | Content Script | `scripting` API | `debugger` (CDP — Chrome DevTools Protocol) |
|------|:-:|:-:|:-:|
| 读 DOM / 提取文本 | ✓ | ✓ | ✓ |
| 模拟点击、滚动 | ✓ | ✓ | ✓ |
| 访问页面 JS 上下文 | ✗ | ✓ | ✓ |
| 后台截图（不切 tab） | ✗ | ✗ | ✓ |
| 模拟 trusted keyboard events | ✗ | ✗ | ✓ |
| 拦截/读取网络请求 | ✗ | ✗ | ✓ |

**为什么必须要 debugger**——没有它反而更吵：

```
无 debugger                           有 debugger
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Agent 需要截图验证操作结果             Agent 需要截图验证操作结果
  → 切到目标 tab（抢焦点）              → CDP Page.captureScreenshot
  → 截图                               → 后台完成，用户无感
  → 切回用户 tab
  → 用户被打断两次                      → 零打断

Agent 需要点击展开内容                 Agent 需要点击展开内容
  → 切到目标 tab（抢焦点）              → CDP DOM.click
  → 执行操作                            → 后台完成
  → 切回                               → 零打断
```

**没有 debugger 的"安静"是假安静**——表面上没有警告栏，但每次 Agent 工作都抢焦点。debugger 的警告栏是一条静态的、用户预期内的提示（"我让 AI 去操作了"），而 Agent 的全部工作都在后台默默完成。

**使用模式：按需 attach，用完 detach**——与 v5 的"用户召唤时行动"一致：

```
用户 ⌘K: "帮我从这个页面提取价格对比"
  → 主 agent 判断需要深度页面访问
  → attach debugger（警告栏出现）
  → Agent 执行：点击展开 → 滚动 → 提取 → 截图验证
  → 完成后 detach（警告栏消失）
  → 结果在画板上呈现
```

**类比 Claude in Chrome**：Claude in Chrome 的用户接受警告栏，因为他们明确知道自己在让 AI 操作浏览器。soma 的场景完全一样——用户通过 chat 召唤 AI 操作网页，debugger 是被召唤的结果，不是默默监控。

**对 ReAct 验证闭环的意义**：§11 描述的 Observe 步骤（Think → Act → Observe）经常需要"看一眼页面状态"——点击后确认是否展开、滚动后确认是否到底、填表后确认是否正确。每一次 Observe 都可能需要截图。没有 debugger = 每次 Observe 都抢焦点。**Agent 越认真验证，用户越被打扰——这与"安静"的产品定位直接矛盾。**

### 权限申请策略：尽早声明，不要后补

Chrome 扩展新增权限时的行为：
1. Chrome 自动**禁用扩展**
2. 用户看到权限变更提示，必须手动选择"重新启用"
3. `debugger` 权限的描述尤其吓人（"Read and change all your data on all websites"）

**用户量越大，后补权限的代价越高**——一部分用户会直接卸载而不是重新授权。

**策略：v0.1 上架时就在 manifest 中声明 `debugger` 权限，同时提供使用该权限的基础功能。**

```
v0.1 manifest.json:
  "permissions": ["debugger", ...]

v0.1 就提供的 CDP 功能（证明权限的合理性）:
  - Clip 增强：动态页面内容提取（SPA、懒加载、tab 后面的内容）
  - 智能截图：clip 时自动截取页面可见区域作为预览
  - 基础页面操作：展开折叠内容、滚动到指定位置

v0.2+ 在已有权限基础上扩展:
  - 主 agent 深度页面操作（翻页、表单、多步提取）
  - ReAct 截图验证闭环
  - 后台多 tab 并行操作
```

**关键决策：宁可在用户少的时候让 5% 的人因权限犹豫不安装，也不要在用户多的时候让 30% 的人因权限变更而流失。** 早期用户对权限的接受度也更高——他们是尝鲜者，愿意给工具更多信任。

**市场背景（2026）**：`debugger` 权限在 AI 浏览器工具中已被广泛采用——Sider、Monica、Claude in Chrome、browser-use 等产品都要求类似权限。用户群体已经形成了"便利大于隐私顾虑"的认知基线，尤其是在明确知道自己在让 AI 操作浏览器的场景中。soma 仍会尽力确保安全和隐私（按需 attach/detach、不常驻监控、不收集非必要数据），但不需要过度担心权限本身吓跑用户。

## 九、三层信号模型（v6 修订）

综合 codex 110 的 Source/Signal/Knowledge + 修正信号的新认识。v6 砍掉 Usage 信号，简化为三层：

### Source（输入信号）

用户的选择：clip、高亮、随手记录。

- **信号含义**："这对我有意义"
- **信号强度**：低（只表达了兴趣，未表达理解）
- AI 的回应：理解内容，初步组织

### Correction（修正信号）

用户对 AI 组织的调整：拖拽重排、改标签、删除、重新连接。

- **信号含义**："你对我的理解这里不对"
- **信号强度**：最高（直接表达了 taste 和思维方式）
- AI 的回应：更新 taste 模型，调整后续行动

### Creation（创造信号）

用户主动写的内容：笔记、观点、方法论。

- **信号含义**："这是我的思考"
- **信号强度**：高（表达了理解和判断）
- AI 的回应：作为最高质量的上下文和 prompt 素材

### 三种信号的关系

```
Source（选择）→ 告诉 AI 什么重要
Correction（修正）→ 告诉 AI 怎么理解用户
Creation（创造）→ 告诉 AI 用户怎么思考

三者共同构成 taste 学习的输入。
```

**v6 简化**：v5 曾增加 Usage 信号（搜索/重访/停留），但这些应用层行为不存在于 CRDT OpLog 中，需要额外的行为日志基础设施，增加了复杂度。v6 砍掉 Usage 信号——taste 学习只依赖 OpLog 中天然存在的三类信号。Correction 信号虽然稀疏，但每一次修正都是高质量的 taste 表达，加上 Source 和 Creation 信号的补充，足够驱动 skill 的迭代。

**与方法论的对应**：
- Source ≈ Think 的第一步（选择）
- Correction ≈ Connect（在修正中发现和建立结构）
- Creation ≈ Think 的深化（用自己的话写）

### Taste 学习（基于 CRDT OpLog + #skill 节点）

v4 的 taste 学习是实时的——每次修正立即调整权重。问题：单次信号噪声大、归因困难。

**原始信号来自 OpLog（零额外采集）**。Loro CRDT 天然记录每一次操作的完整历史（谁改的、什么时候、改了什么），这就是最准确的修正信号——不需要额外的行为日志基础设施。

**派生规则存为 #skill 节点的子节点（可审查可修正）**。Schema evolution skill 定期扫描 OpLog，用 LLM 分析多次一致的修正模式，派生出规则，写入对应的 #skill 节点。用户可以在 Schema 中看到这些规则，手动修改或删除。

```
Loro OpLog（原始信号，零额外采集）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  agent@T2: node_abc.tags: +#AI（agent clip 时加的）
  user@T3: node_abc.tags: -#AI, +#Product Design（用户改了）
  ...重复 3 次同模式...

       ↓ Schema evolution skill 扫描 OpLog（LLM 分析修正模式）→ 更新 #skill 节点

#skill: clip 文章结构提取
━━━━━━━━━━━━━━━━━━━━━━━
  Rules（子节点，从 OpLog 修正中派生）
    ├── 设计类内容不标 #AI → 标 #Product Design
    │     #rule | Source: correction | Hit Count: 3
    ├── 不生成 "Core argument" 字段
    │     #rule | Source: correction | Hit Count: 2
    └── ...
  用户可以直接编辑这些规则 ← 比任何隐式信号都强
```

**为什么 OpLog 作为原始信号源**：
- **零额外采集**：OpLog 是 CRDT 的核心数据结构，已经存在
- **零图谱污染**：不需要 `#behavior-log` 节点混入用户的知识图谱
- **最准确**：OpLog 记录的是实际发生的操作，不是二次转述
- **天然区分操作者**：每个 op 标记了是 user 还是 agent 发起，修正信号的归因是确定性的

**派生规则为什么存为节点**：
- **可审查**：用户可以看到 AI 学到了什么
- **可修正**：用户直接编辑规则，比等 AI 从 OpLog 中学到更快更准
- **一切皆节点**：taste 不是独立的系统，只是 #skill 节点的子节点

**安全边界**：
- 衰减：旧信号权重随时间降低（偏好会变）
- 置信度阈值：需要多次一致信号才调整，单次行为不过拟合

| | 实时学习（v4） | OpLog + #skill 节点（v6） |
|---|---|---|
| 信号可靠性 | 低（单次行为噪声大） | 高（多次行为才形成模式） |
| 原始信号存储 | N/A | 零额外采集（OpLog 天然存在） |
| 派生规则存储 | 隐式（不可见） | #skill 子节点（可审查可修正） |
| 图谱污染 | N/A | 零（不创建行为日志节点） |
| 误学习风险 | 高（过拟合单次修正） | 低（需多次一致信号 + 用户可直接改） |
| 用户感知 | 不知道 AI 在学什么 | 可以看到和修改 AI 学到的规则 |

### Review（回顾环的核心：认知镜像）

Review 是三条认知工作流中**回顾环**的核心机制。它独立于 taste 学习——taste 学习优化的是 **soma 怎么工作**（基础设施），Review 照亮的是**用户的认知怎么在演化**（核心价值）。

**Review 是完全可选的。** 轻度用户可能永远不会用 `/review`——阅读环和碰撞环不依赖 Review。Review 是给那些想要看见自己认知演化的用户准备的，类似于投资者的"年度回顾"——不是总结做了什么，而是审视自己的判断框架怎么变了。

#### Review 的两个层次

**层次 1：内容脉络**（表层）——"你做了什么"

```
用户: /review this week

Agent: "这周你写了 8 条笔记，clip 了 3 篇文章。
       RAG 出现了 4 次，你和老王聊了两次都涉及搜索优先级。
       有 1 篇 fine-tuning 文章 clip 后没有后续笔记。"
```

这是基础的内容摘要，有用但不深。

**层次 2：认知镜像**（深层）——"你的认知发生了什么"

```
用户: /review this month

Agent: "几个认知结构层面的变化——

1. **新结构浮现**：你这个月开始用'干/湿'框架思考分工问题——
   它出现在 3 篇不同主题的笔记中（AI 协作、团队管理、阅读方法），
   可能正在成为你的一个通用思考工具。

2. **旧结构升级**：你对'RAG vs fine-tuning'的理解从
   二选一（Week 1: '也许应该先做 RAG'）演化为
   互补（Week 3: '不同场景用不同策略'）。
   你自己注意到这个转变了吗？

3. **未解决的张力**：你的笔记中'快速迭代'和'深度思考'
   反复出现矛盾——3 月 5 日你写'要快'，3 月 8 日你又写
   '慢下来想清楚再动手'。这个张力还没有被你显式地面对过。

4. **隐藏的同构**：你 3 月 2 日写的关于代码重构的笔记
   和 3 月 9 日写的关于信息源管理的笔记，结构上惊人地相似——
   都在说'约束带来自由'。你可能在不同领域反复发现同一个模式。"

用户: "4 太有意思了，我完全没注意到。帮我把这两篇放在一起，
       我想写一篇关于'约束与自由'的思考。"
```

**层次 2 的关键**：Review 不是在告诉用户"你做了什么"，而是**让用户看见自己看不见的东西**——新结构的浮现、旧结构的升级、未解决的矛盾、跨领域的同构。这是认知镜像，不是活动摘要。

#### Review 的价值分层

Review 不应该把所有内容平铺给用户——不同信息需要的注意力不同。AI 在生成 Review 时自动做价值分层：

| 信号 | 含义 | Review 中的处理 |
|------|------|---------------|
| `#note` 标签 | 用户亲手写的思考 | 高权重，深度回顾 |
| `#spark` 标签 + 用户追加了子笔记 | AI 触发且用户响应了 | 高权重，关注用户响应的部分 |
| `#spark` 标签 + 无用户互动 | AI 触发但用户没回应 | 低权重，快速提及或跳过 |
| 被多次引用/碰撞命中 | 高连接密度 | 值得深度回顾，可能是核心思维模式 |
| 长期未触碰的笔记 | 沉睡内容 | 偶尔提起，看是否值得重新审视 |

价值分层是 AI 的内部能力，不是用户面对的选择。用户不需要标记"这条重要"——AI 从标签、连接密度、互动信号中自动判断注意力分配。

#### Review 的本质

Review 的本质不只是 Connect——它是**让用户看见自己的取景框在怎么变形**。

用户每天在自己的认知里，看不到全貌。Review 把用户的隐性认知演化摆到桌面上：你以为你这个月在研究 A，但 Review 告诉你其实你真正在意的是 B，而且 B 和你三个月前放弃的 C 之间有一个你没注意到的同构。

这正是回顾环的闭合点：用户看见了自己的认知轮廓 → 产生新的反思 → 反思改变了下次阅读和思考的方向。

#### Review vs Taste 学习

| | Taste 学习（基础设施） | Review（核心价值） |
|---|---|---|
| **触发** | Schema evolution skill 定期扫描 OpLog | 用户主动（`/review`） |
| **目的** | 让 AI 的「干」越来越准（基础设施） | 让用户看见自己的认知演化（核心价值） |
| **对象** | 修正模式（标签修正、字段偏好） | 认知结构变化（新结构、升级、矛盾、同构） |
| **数据来源** | CRDT OpLog（原始修正信号） | 节点图谱（用户的全部内容） |
| **输出** | 更新 #skill 节点中的规则 | 认知镜像，用户可继续行动 |
| **用户感知** | 无感（工具自己变好了） | 有感（"原来我一直在想这个"） |
| **与方法论的关系** | 工具优化 | Connect 的最高形式——与自己的连接 |

## 十、AI 交互设计

> v4 的交互设计试图让 AI 在每次 Enter 时自动结构化，通过 Ghost UI 呈现。交互分析发现这在 outliner 场景中不可行——快速连续输入时 ghost UI 造成布局跳动和注意力打断，300px 侧边栏中 ghost fields 挤占空间。v5 重新设计了 AI 的介入方式。

### 核心原则：输入通道分离，输出画板共享

```
┌────────────────────────────────────────────────────────┐
│                    Outliner（画板）                      │
│            用户和 AI 的共享输出面                         │
│            所有结果都是节点，形式一致                      │
└────────────────────────┬───────────────────────────────┘
                         │
         ┌───────────────┼───────────────┐
         │               │               │
    ┌────┴────┐    ┌─────┴─────┐   ┌─────┴─────┐
    │ Node    │    │ Chat      │   │ Clip      │
    │ 输入    │    │ 指令      │   │ 认知触发  │
    │         │    │           │   │           │
    │ 用户    │    │ 用户→AI   │   │ 用户→AI   │
    │ 直接画  │    │ 说了再做  │   │ 帮我看看  │
    │         │    │           │   │           │
    │ AI 不   │    │ AI 默认   │   │ AI 提取   │
    │ 介入    │    │ 执行      │   │ 结构+碰撞 │
    └─────────┘    └───────────┘   └───────────┘
```

### 三个 AI 介入通道

**通道 1：Node 输入 — AI 不介入**

用户在 outliner 中写什么，原样保留。这是用户的思考空间。

唯一的 AI 辅助是已有的**规则匹配**（非 LLM）：
- 输入 `@` → 已有节点名模糊匹配（毫秒级）
- 输入 `#` → 已有标签模糊匹配（毫秒级）
- 日期文本 → 正则识别（"明天"、"下周一"）

这些是确定性的、即时的、不调用 LLM 的。outliner 保持快速和可预测。

**通道 2：Chat — 独立面板，执行→报告→反馈**

Chat 是用户与 AI 协作的主要界面——独立面板，对话持久化，支持多轮。

**Chat 的两种模式**

Chat 面板承载两条认知工作流的不同侧面：

**模式 1：执行模式（基础设施）— 执行→报告→反馈**

```
用户: "帮我整理一下今天记录的笔记"
  ↓ 主 agent 直接在画板上操作（LoroDoc 修改）
AI: "我做了以下整理：
     · 给 3 条笔记加了 #meeting 标签
     · 把 2 条 AI 相关的归到一起
     · 创建了 People 字段并填充了提到的人名
     画板上已经更新了，你看看？"
  ↓ 用户三种反馈路径
用户: "好" → 对话继续（修改保留）
用户: "#meeting 应该是 #1on1" → AI 调整 → 再次报告
用户: "撤回" → AI undo 整个操作（单次 CRDT commit = 单次 undo）
```

类比 Claude Code——直接改文件，报告做了什么，用户可以说"不对，改回去"。

**模式 2：碰撞模式（碰撞环的核心）— 想法→加密→方向**

```
用户: "我刚看到一篇文章说约束反而带来创造力，
       让我想到我们代码重构时也是这样——限制越多反而设计越干净。
       这是不是一个更普遍的模式？"

AI: "有意思。你的直觉可能指向一个跨领域的结构——

     在软件中，接口约束 → 实现自由（依赖倒置原则）。
     在写作中，格律约束 → 表达自由（十四行诗 vs 自由诗）。
     在你上周写的信息源管理笔记里，你也提到了
     '关注 10 个公众号反而比关注 100 个看得更多'。

     这三者的同构是：下一层的约束消除了上一层的选择焦虑，
     释放了注意力给真正重要的事。

     不过有一个张力：你 3 月 5 日还写过'不要过早抽象'——
     这也是一种约束，但方向相反。两种约束在什么条件下各自适用？"

用户: "这个区分很重要。一个是减少选项的约束，
       一个是推迟决策的约束。帮我把这个想法记下来。"
```

碰撞模式的关键区别：
- **输入**不是"请 AI 做什么"，而是用户自己的初生想法
- **AI 的角色**不是执行者，而是思维的加密器——多角度碰撞、拉高密度、找同构/冲突
- **输出**不是结论，而是更清晰的问题和新的方向
- **环闭合**：用户带着更致密的想法离开，下次阅读时的取景框已经不同了

碰撞模式下 AI 能引用用户图谱中的已有笔记作为碰撞素材——这是 soma 的独特优势：AI 不只有通用知识，还有用户自己的思考历史。

**Chat 面板的特性**：

- **独立面板**：在多面板布局中与 outliner 并列（宽屏）或作为抽屉覆盖（窄屏）。详见 `docs/research/multi-panel-design.md`
- **对话持久化**：历史对话可回顾（"上周我让 AI 整理过什么？"），独立于节点图谱存储（Chat 有自己的持久化层，不混入 outliner）
- **多轮对话**：AI 保持上下文，用户可以连续指令（"再把那几条按时间排序" → AI 知道"那几条"指上一轮操作的节点）
- **⌘K 快捷打开**：⌘K 打开 Chat 面板（或聚焦已打开的 Chat），也可点击工具栏入口
- **画板联动**：Chat 中提到的节点可点击跳转到画板位置；用户在画板上选中节点后打开 Chat，AI 自动识别选中的上下文
- **后台任务指示器**：Chat 面板右上角显示后台任务入口（badge 标注运行中数量）。点击展开任务列表——每个任务显示名称、状态（运行中 / 已完成 / 失败）和结果链接。用户与主 agent 的对话不受后台任务阻塞

**为什么不是 Command Palette 的扩展**：早期设计认为 Chat 是"⌘K 命令面板 + 自然语言"，指令是临时的。实际上 Chat 需要：多轮上下文（"再改一下"）、执行报告（告诉用户做了什么）、反馈闭环（用户说"不对"）、历史回顾（"上次怎么整理的"）——这些都需要独立的持久化面板，Command Palette 承载不了。⌘K 仍然是快捷键，但打开的是 Chat 面板。

**Raycast 统一入口模型**：参考 Raycast 将搜索框和 AI Chat 统一为一个入口的设计。soma 的 ⌘K 是统一入口——输入框同时支持命令搜索和自然语言对话：

```
⌘K 打开统一输入框
  │
  ├── 输入 "/" → 命令面板模式（快捷操作列表）
  ├── 输入节点名 → 搜索模式（fuzzy match 跳转）
  └── 输入自然语言 → Chat 模式（输入框自动扩展为 Chat 面板）
      └── Tab 键 → 展开完整 Chat 面板（多轮对话、历史）
```

统一入口的好处：用户不需要区分"我是要搜索还是要跟 AI 说话"，输入即可。系统根据输入内容自动判断模式。Chat 面板从输入框自然生长出来，而不是一个独立的入口——降低了"打开 AI"的心理门槛。

**通道 3：Clip & Spark — 认知触发，不是剪藏**

> **2026-03-11 重新定位**：Clip 从"剪藏"（保存原文）重新定位为"认知触发"（提取结构 + 碰撞已有笔记 + 邀请用户写想法）。灵感来源：李继刚的阅读实践——"读书变成了启发我思考的机制，提取结构交给 AI，我去找灵感找启发"。

用户按下 Clip，不是在说"帮我存一下"，而是在说**"帮我看看这篇东西"**。AI 的任务不是保存原文，而是：

1. **提取结构**（读薄，始终执行）：这篇内容的核心思维框架/取景框是什么？不是"要点 1234"，而是"作者用了什么结构来思考这个问题"。**这是 Spark 的核心价值，从 Day 1 就是差异化体验**
2. **碰撞已有笔记**（读厚，有置信度阈值）：AI 通过 `node.search` + `node.read` 在图谱中渐进式发现相关节点，找到关联、冲突或同构。**碰撞是 pull 不是 push——只在置信度高时才展示，宁可不碰不硬碰。** 密度不够时（如冷启动期）Spark 只呈现结构提取，不强行凑碰撞
3. **邀请涟漪**：呈现结构 + 碰撞结果（如果有）后，留白——用户写下自己的想法

**原文不存储**——URL 即回溯路径，AI 在 clip 时一次性完成理解（Shadow Cache + TTL），理解结果（结构 + 碰撞）留下，原文可过期。

**Clip 的输出形态**：

碰撞有置信度阈值——密度不够时只展示结构提取 + 留白，不强行凑碰撞：

```
冷启动期（碰撞素材不足）：

用户 clip 一篇关于"模块化建筑"的文章
  ↓
AI 提取结构：
  "核心框架：预制约束 → 组装自由。决策提前锁定在模块边界上，
   换来现场组装的灵活度和速度。"

留白：
  [光标停在这里——这让你想到什么？]
```

```
积累期（碰撞置信度过阈值时自然出现）：

用户 clip 一篇关于"模块化建筑"的文章
  ↓
AI 提取结构：
  "核心框架：预制约束 → 组装自由。..."

碰撞已有笔记（置信度高，展示）：
  "你 3 月 2 日写的关于 API 设计的笔记也在说类似的事——
   '接口越窄越好用'。两者的同构是：下一层约束带来上一层自由。
   但你上周还写过'不要过早抽象'——这个约束方向不同，值得区分。"

留白：
  [光标停在这里——这让你想到什么？]
```

**处理逻辑是进化的 #skill，不是固定模板**：

Clip 处理不是一段硬编码的 prompt，而是一个 #skill 节点——通过 OpLog 修正模式持续进化，因人而异、因内容而异：

```
#skill: 内容结构提取
  Rules（从用户修正中进化）
    ├── 提取思维结构/取景框，不罗列要点
    ├── 论文：额外提取方法论框架和核心假设
    ├── 个人博客：关注作者的推理路径和隐含信念
    ├── 定价页：提取比较矩阵和定价模型结构
    ├── 碰撞阈值：至少 2 条已有笔记强关联才展示
    └── ...（从用户修正中持续生长）
```

每个用户的这个 skill 最终长出来的形状都不同——那就是每个人的 taste。避免了"兵马俑式"的固定模板降气。

**Clip 的价值渐进增长**（碰撞飞轮）：
- **Day 1**：Spark = 高质量结构提取 + 留白。**结构提取本身就是差异化**——不是 AI 摘要（要点 1234），而是提取作者的思维框架/取景框。碰撞尚无素材，不强行展示
- **Week 2+**：碰撞开始零星出现——当置信度越过阈值时，Spark 中自然展示与近期笔记的关联
- **Month 2+**：碰撞质量提升——"这篇文章的框架和你上周写的笔记有同构"
- **Month 6+**：碰撞成为 clip 的主要价值——用户 clip 不是为了存，是为了看撞出什么
- **碰撞的主要入口始终是用户主动发起的**：Chat 碰撞模式（从 Day 1 可用，不依赖笔记积累）和 `/review`（用户有预期，薄连接也不反感）

**元数据仍然提取**（is / has / about 原则不变）：

| 层次 | 含义 | AI 做什么 | 示例 |
|------|------|----------|------|
| **is**（supertag） | 这个节点是什么类型 | 识别类型，应用已有 supertag | `#article` |
| **has**（field） | 这个节点有什么属性 | 按 supertag 模板提取，直接填充 | `Author: Simon Willison` |
| **about**（引用） | 这个节点关于什么主题 | 如有 Topics 引用字段，填充引用 | `Topics: RAG →` |

但元数据是**基础设施层**（让节点可搜索、可过滤），不是 clip 的核心价值。核心价值是**结构提取 + 碰撞**。

**Clip 不做的事**：
- 不存储原文（URL 即回溯路径）
- 不发明新的 supertag（从多次模式中涌现，不是 clip 时临时决定的）
- 不用固定模板处理所有内容（处理逻辑是进化的 #skill）
- 修正进入 OpLog → #skill 迭代 → 下次更准

**命名：Clip & Spark**。Clip = 采集动作（用户熟悉的"我要这个"），Spark = 激发体验（soma 独有的结构 + 碰撞结果）。UI 上按钮叫 **Clip**，点击后展开的结果面板叫 **Spark**——显示结构提取 + 碰撞结果 + 留白。火花是碰撞的结果（包含了 collide 的含义），是起点不是终点（邀请用户写下想法）。产品语言延伸：Think where you read. **See what sparks.**

**Spark 触发策略**：

| 场景 | Clip 创建方式 | Spark 触发 | 理由 |
|------|-------------|-----------|------|
| 用户主动 clip | `saveWebClip()` | **自动** | 明确意图："帮我看看这个" |
| 用户先 highlight/note，被动创建 clip | `createLightweightClip()` | **手动按钮** | 用户在思考中，不要打断 |
| 没有 API key | 任意 | **按钮灰显/提示设置** | 不阻塞其他功能，以后再用 |

Spark 按钮在 #source 节点上**始终可见**——已经自动 Spark 过的可以再次触发（比如用户后来又加了新 highlight，想重新碰撞）。

**Clip & Spark 节点结构**：

> **设计灵感**：参考李继刚「深度拆书机」的三轮认知压缩（骨架→血肉→灵魂）和 Epiplexity 原理（可学习的结构 vs 噪声的切割线取决于投入的算力）。三轮自然映射到 soma 的节点树 + 碰撞环。

Spark 不是表单（固定字段），而是一棵**有机的思维树**。骨架是顶层结构（一眼看清），血肉是骨架的 children（展开看深），灵魂是碰撞环的事（跨内容连接，随积累渐进出现）：

| 轮次 | 载体 | 时机 | 对应 |
|------|------|------|------|
| **Round 1 骨架** | #spark 顶层子节点 | clip 时自动 | "这篇在说什么"——核心框架/取景框 |
| **Round 2 血肉** | 骨架的 children（展开查看） | clip 时自动 | "凭什么这么说"——论证链、隐形假设、边界条件 |
| **Round 3 灵魂** | 碰撞环（Chat / Review / Spark 碰撞段） | 积累后渐进 | "还能怎么用"——跨域映射、与已有知识连接 |

**outliner 天然支持这种渐进式披露**——骨架扫一眼，想看深就展开血肉，跨内容连接通过碰撞环在使用中自然生长。不需要任何特殊 UI。

```
Journal / Today
  └── "Modular Architecture in Practice" #source (#article)
        ├── Highlights (field, 默认折叠)
        │     ├── "预制约束决定了组装的自由度…"
        │     ├── "模块边界应由变化率决定…"
        │     └── "接口越窄，上层越灵活…"
        ├── #spark                                    ← AI 认知触发（持久化节点树）
        │     ├── 核心框架：预制约束 → 组装自由        ← 骨架（Round 1，顶层一眼看清）
        │     │     ├── 论证：模块边界由变化率决定      ← 血肉（Round 2，展开看深）
        │     │     │     └── → ref → "模块边界应由变化率决定…"
        │     │     ├── 隐形假设：假定变化率可预测
        │     │     └── 边界：高度耦合系统中模块化反而增加成本
        │     ├── 论证结构：归纳型（案例→模式→原则）    ← 骨架
        │     └── 碰撞（置信度过阈值时出现）：           ← 灵魂（Round 3，碰撞环）
        │           └── 跟 [[API 设计笔记]] 是同构 → ref
        ├── #note                                     ← 用户笔记
        │     └── 约束有两种：边界约束和时序约束… → ref → highlight
        └── (URL / Author / Published — field entries)
```

关键设计决策：
- **#source 是顶层锚点**：来源元数据（URL、作者、日期）作为 field
- **#highlight 是 #source 的 Highlights 字段**：默认折叠，不干扰阅读。highlight 由用户选中或 AI 处理时提取，共享引用池
- **#spark 和 #note 是 #source 的平级子节点**：AI 的认知触发和用户的笔记自然共存，不做视觉区分
- **Spark = 骨架 + 血肉（树状渐进披露）**：骨架是 #spark 的顶层子节点，血肉是骨架的 children。用户不展开就只看骨架（取景框），展开就看论证链、隐形假设、边界条件。不同 #skill 产出不同深度的树
- **碰撞段在 Spark 树内有条件出现**：碰撞结果（Round 3 灵魂）仅在置信度过阈值时作为 #spark 子节点出现，用 inline reference 物化为图谱边。碰撞的主要入口仍是用户主动的 Chat 碰撞模式和 /review
- **多深度处理**：不同内容类型自动匹配不同 Spark 深度——短文 = 骨架为主，长文/研究 = 骨架 + 完整血肉，由 #skill 规则决定
- **Spark 持久化为真实节点**：可搜索、可引用、可被后续碰撞命中。AI 生成的 Spark 节点和用户手写的 note 是平等公民
- **用户可在 Spark 内部追加笔记**（子节点）、也可在 Spark 旁边写独立 note（兄弟节点），边界自然模糊

三层可独立出现：只有 #source（快存）、#source + #spark（AI 处理了）、#source + #note（用户直接写了）、三者都有（完整认知循环）。

**Review 中的价值分层**：详见 §9「Review 的价值分层」。AI 在生成 Review 时自动做价值分层——基于标签（#spark/#note）、连接密度、互动信号分配注意力权重，引导用户关注最值得花时间的内容。

**通道 4：Tag/Field 创建 — AI 给建议**

用户在定义结构时（创建 tag、添加 field），AI 提供建议：

- 创建 `#meeting` tag → AI 建议字段：People, Topic, Decision
- 添加 field → AI 建议类型和 options
- 建议以 Ghost UI 呈现：灰色虚线，点击接受，忽略消失

**Ghost UI 只在 tag/field 创建时出现**——clip 不再使用 Ghost UI（直接填充），node 输入 AI 不介入。Ghost UI 的范围收窄到"用户在定义结构"这一个时刻。

### 四层干预模型（v6 修订）

v4 的四层模型保留，但各层的内容因输入分离而变化：

```
Layer 0: 完全不可见
  ├── Schema evolution skill 扫描 OpLog → LLM 分析修正模式 → 更新 #skill 节点
  └── 网页内容预分析

Layer 1: 被动可见（用户不操作时静默，操作时浮现）
  ├── 搜索结果智能排序
  └── 折叠节点摘要

Layer 2: Ghost UI（仅在 tag/field 创建时出现）
  ├── 创建 tag 时的字段建议（Ghost field）
  └── 创建 field 时的类型/options 建议
  （Clip 不再使用 Ghost UI——直接填充，画板可见可修正）

Layer 3: 主动对话（Chat 指令 + Review）
  ├── Chat 指令执行结果
  ├── Review 对话（用户发起：/review this week）
  └── 危险拦截确认（批量删除等）
```

**关键变化**：Layer 2 从"随处出现的 Ghost UI"收窄为"仅在定义结构时出现"。Layer 3 从"审批式确认"变为"对话式协作（chat）+ 用户发起的 Review（内容反思）"。

### AI 行动时机

| 时机 | 触发条件 | soma 中的应用 |
|------|---------|-------------|
| **实时** | `@` / `#` 输入 | 规则匹配，毫秒级，不调 LLM |
| **用户召唤时** | Chat 指令（⌘K） | LLM 理解指令 → 在画板上执行 |
| **主动 clip 时** | 用户点击 Clip | 自动 Spark：LLM 提取结构 + 碰撞 → #spark 节点树 |
| **手动 Spark** | 用户点击 Spark 按钮 | 对已有 #source 补充/重新生成 Spark（被动 clip 或新增 highlight 后） |
| **定义结构时** | 创建 tag / field | LLM 建议字段/类型 → Ghost UI |
| **用户发起** | `/review` 命令 | Review 照亮认知演化（认知镜像 + 价值分层） |

**v5 的关键教训**：**可预测性不是"AI 多快"的问题，而是"AI 在哪里出现"的问题。** v4 试图让 AI 在每次 Enter 时可预测地出现——但 outliner 的输入节奏本身就不可预测（有时连续 10 行，有时停下来想 30 秒）。v5 把 AI 的出现绑定到明确的用户动作上——chat/clip/tag 创建——时机完全可预测。

## 十一、Agent 系统架构（v6 简化）

> **核心原则：一切皆节点。** agent 的身份、行为规则、技能、记忆——全部是节点。agent 读节点知道自己是谁、能做什么、怎么做；用节点记录结果和学到的规则；用户改节点改变一切。唯一不是节点的是 Tools（代码层）。

### 架构概览

```
节点图谱（唯一的数据层）
━━━━━━━━━━━━━━━━━━━━━━
  ├── soma（#agent，主 agent 身份定义，用户可改）
  │     Role: 理解用户意图，在画板上执行，委派复杂任务
  │     Tools: node, browser, undo
  │
  ├── #skill 节点们（agent + 用户共同迭代）
  │     ├── soma 核心行为准则（Always Active，产品预置，用户可改）
  │     │     ├── 安静融入，非必要时静默，等待召唤
  │     │     ├── 不修改用户原文
  │     │     ├── 不确定就留空，不虚构
  │     │     └── ...
  │     ├── 深度网页操作和内容提取
  │     ├── 从定价页提取 LLM 模型信息
  │     ├── 整理会议笔记
  │     ├── Schema evolution（从 OpLog 修正中迭代 skill 规则）
  │     └── ...（AI 涌现的 / 用户手写的）
  │
  └── 其余所有节点 = agent 的记忆 = 用户的知识
        ├── Schema（tagDef / attrDef）
        ├── 日期节点 + 用户内容
        ├── Settings
        └── ...

Tools（代码层，唯一不是节点的东西）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ├── node (create|read|update|delete|search)
  ├── browser (16 actions: read_page|click|type|...)
  └── undo (撤回上一步操作)

运行时（不持久化，用户不可见）
━━━━━━━━━━━━━━━━━━━━━━━━━━━
  └── 主 agent 按需创建 subagent（临时角色，上下文隔离，用完即弃）
```

**v5→v6 的关键简化**：

| v5 | v6 |
|---|---|
| 五层架构（SOUL/CONTEXT/MEMORY/SKILLS/TOOLS） | 节点 + Tools |
| SOUL = 硬编码产品逻辑，用户不可见不可改 | #agent 节点 + Always Active #skill 节点，用户可查看可修改 |
| MEMORY = 独立的 Taste Model + Skill States | agent 的记忆 = 整个节点图谱 |
| 预定义多 agent（Chat/Web/Connect/Schema Agent） | 主 agent（#agent 节点）+ 运行时临时 subagent |
| locked skill / 元规则不可自改 | 没有 locked，一切可改 |

### 为什么只有节点和 Tools

同一份节点图谱，用户看到知识，agent 看到配置：

- **#agent 节点** = agent 的身份定义（"我是谁"）
- **#skill 节点** = agent 的行为规则和执行能力（"怎么为这个用户工作"）
- **Schema 节点** = 数据结构定义（"节点长什么样"）
- **其余节点** = agent 的记忆 = 用户的知识（"我知道什么"）

agent 启动时读节点构建上下文：加载 #agent 定义 → 加载 Always Active #skill → 按任务匹配其余 #skill → 通过 node.read/search 访问相关知识。**没有独立的 MEMORY 层，也没有独立的 CONTEXT 层——图谱本身就是一切。**

### #agent 与 #skill

**#agent** = 主 agent 的身份定义（一个）。定义了 agent 是谁、做什么、用什么工具。产品预置 soma 主 agent 节点，用户可以修改它的行为描述。

**#skill** = 能力和规则（多个）。定义了 agent 怎么做具体的事。区别只在属性：

| 属性 | 含义 | 示例 |
|------|------|------|
| Always Active | 每次 agent 执行都加载 | soma 核心行为准则 |
| 按需匹配 | 根据当前任务相关度加载 | 从定价页提取 LLM 信息 |

所有 #skill 节点对 agent 和用户的权限一样——agent 可以迭代（从 OpLog 修正中派生新规则），用户也可以直接编辑（"别再给设计类内容标 #AI 了"）。没有 locked skill。

**Schema ≠ Skill**（保持 v5 的区分）：Schema（supertag/field 定义）= 数据结构（what），Skill = 执行能力（how）。一个 Skill 可能引用 Schema 作为其输出模板（"提取结果按 #llm-model 的字段填充"），但 Skill 本身包含的是执行逻辑。

### Skill 节点结构

Skill 就是被 #skill 标记的普通节点，用节点树的方式查看、编辑、删除：

```
#skill: 从定价页提取 LLM 模型信息
  Type: extraction
  Output Schema: #llm-model →        ← ref 指向 tagDef（输出模板）
  Trigger: URL matches */pricing*
  Confidence: 0.87                   ← agent 自维护
  ─────────────
  Rules                              ← 子节点：执行规则（agent + 用户共同迭代）
    ├── 价格格式: "$X / 1M tokens"
    │     #rule | Source: inferred | Hit Count: 14
    ├── Context 单位总是 K
    │     #rule | Source: inferred | Hit Count: 8
    └── Google 用 "characters" 不是 "tokens"
          #rule | Source: correction | Hit Count: 3
  Failures                           ← 子节点：失败案例
    └── 2026-03-08: Google 价格单位错误
          #skill-failure
```

**Skill 类型**：extraction（数据提取）、organization（分类组织）、connection（关联发现）、verification（自检验证）。

### Skill 迭代（agent + 用户共同进化）

Skill 不是静态指令，而是 **agent 和用户共同迭代的活文档**。

**两条迭代路径**：

1. **Agent 迭代**：Schema evolution skill 扫描 CRDT OpLog 中的修正模式，用 LLM 分析 → 多次一致信号 → 更新 #skill 节点的 Rules 子节点

```
CRDT OpLog（原始修正信号，零额外采集）
  ├── user 修正了 agent 加的标签/字段
  ├── user 忽略了 Ghost 建议
  └── agent 自检发现异常

       ↓ Schema evolution skill 扫描 + LLM 分析修正模式 → 更新 #skill 节点

#skill 规则更新（以子节点形式记录，可审查）
  · Google 价格单位改了 3 次 → 新增 rule: "用 characters 不是 tokens"
  · #decision 标签建议 5 次、0 次接受 → 新增 rule: "不再建议 #decision"
  · 用户总是接受 People 字段 → Confidence 提升
```

2. **用户迭代**：用户直接编辑 #skill 节点——增删改规则、调整触发条件、修正失败案例。比等 agent 从 OpLog 中学到更快更准。

**安全边界**：
- **衰减**：旧信号权重随时间降低（偏好会变）
- **多次一致信号**：单次修正不触发调整，需多次同方向信号才更新
- **进化审计**：每次调整记录为子节点（可回溯）

### Subagent（原子能力的运行时编排）

v5 预定义了四个 agent（Chat/Web/Connect/Schema），v6 简化为：**只有一个持久化的主 agent（#agent 节点），subagent 是运行时按需创建的临时角色。**

**#skill 是原子能力，subagent 是原子能力的临时组合。** 主 agent 可以直接使用任何 #skill，也可以把一组相关 skill 编排为一个 subagent 委派执行：

```
#skill（原子能力，持久化在图谱中）
  ├── 提取页面文本
  ├── 识别价格格式
  ├── 创建结构化节点
  ├── CDP 截图验证
  └── ...

subagent = 原子能力的异步编排（运行时创建，后台执行，完成后销毁）
  例："从定价页提取所有模型"
    = 提取页面文本 + 识别价格格式 + 创建结构化节点 + CDP 截图验证
    在隔离上下文中后台执行，完成后结果写入画板 + 通知主 agent

main agent = 可以使用所有原子能力
  ├── 简单任务 → 直接调用 #skill（同步，Chat 中即时响应）
  └── 复杂任务 → 编排为 subagent 后台委派（异步，不阻塞 Chat 对话）
```

**关键设计：Chat 永不阻塞。** 主 agent 接到复杂任务 → 判断需要拆分 → 创建 subagent 后台执行 → **立即回到 Chat 继续和用户对话**。subagent 在后台独立运行，完成后结果写入画板（节点），主 agent 在 Chat 中报告完成。用户在等待期间可以继续聊天、发起新任务、甚至让主 agent 同时启动多个 subagent。

```
用户: "帮我从 Anthropic、OpenAI、Google 三个定价页收集模型信息"
  → 主 agent: "好的，我在后台处理了。" → 创建 subagent（后台）
  → Chat 面板右上角出现 badge (1)

用户: （继续和主 agent 聊别的）"刚才那篇文章让我想到一个问题…"
  → 主 agent 正常响应碰撞模式对话（不受后台任务影响）

  → subagent 后台完成 → 结果写入画板 → badge 更新为 ✓
  → 主 agent 在 Chat 中插入通知："LLM 定价对比已完成，创建了 8 个 #llm-model 节点。[查看]"
```

**后台任务指示器**：Chat 面板右上角显示运行中的任务数量（badge）。点击展开任务列表——每个任务显示名称、状态（运行中 / 已完成 / 失败）和结果链接。用户可以取消运行中的任务（主 agent 终止对应 subagent）。

**用户视角**：只有 #skill 节点和一个 #agent 节点。编排和委派是 agent 的运行时决策，用户看到的是"主 agent 帮我安排了一个后台任务"。
**运行时**：主 agent 按需创建 subagent 后台执行。简单任务同步完成（Chat 中直接报告），复杂任务异步委派（后台执行 + 完成通知）。
**CRDT 天然支持并发**——不同 subagent 操作不同节点时完全并行，操作同一节点时 CRDT 自动合并。多个 subagent 可同时运行。

### 多 Agent 通信架构

> 详细调研见 `docs/research/multi-agent-orchestration.md`。调研了 Google ADK、OpenAI Agents SDK、LangGraph.js、KaibanJS、Vercel AI SDK 6、CrewAI、AutoGen、Mastra、Claude Code Agent Teams 等 10+ 方案，结论：**没有现成框架满足我们的约束**（浏览器单进程 + CRDT 数据层 + mid-task clarification + 轻量 bundle）。在 pi-agent-core 上自建 AgentOrchestrator。

**双层通信模型**：

```
Loro CRDT（Blackboard — 数据交换）    消息总线（EventTarget — 协调信号）
  ├── 结果节点（subagent 写入画板）      ├── task-delegated（主 → sub）
  ├── 进度节点（可选）                   ├── task-completed / task-failed（sub → 主）
  └── 协作数据（Agent A → Agent B）     ├── clarification-needed / response（sub ↔ 主）
                                         ├── task-cancelled（主 → sub）
                                         └── data-available（sub → sub）
```

- **数据交换通过 Loro CRDT**：结果、进度、协作产出全部是节点（一切皆节点）。Agent A 写 URL 节点 → Agent B 通过 CRDT subscription 监听并处理
- **协调信号通过 EventTarget 消息总线**：任务生命周期、求助、取消等轻量事件。EventTarget 是浏览器原生 API，零依赖

**四种通信场景**：

| 场景 | 方向 | 机制 |
|------|------|------|
| 求助 | subagent → 主 agent | 消息总线 `clarification-needed` → 主 agent 自行判断或转问用户 → `clarification-response` → subagent 恢复 |
| 完成通知 | subagent → 主 agent | 结果写入 Loro + 消息总线 `task-completed` → 主 agent 在 Chat 中报告 |
| 取消 | 主 agent → subagent | pi-agent-core `abort()` + 消息总线 `task-cancelled` |
| 协作 | subagent ↔ subagent | Loro CRDT 节点写入 + subscription 监听（数据管道）；复杂协调用消息总线 `data-available` |

**求助流（关键差异化能力）**：

```
Subagent 执行中遇到歧义
  → 消息总线: clarification-needed { question: "页面有两个价格表，用哪个？" }
  → subagent 状态: running → waiting-clarification
  → 主 agent 收到 → 判断：
      ├── 自行回答（有足够上下文）
      └── 转问用户（在 Chat 中显示："后台任务需要确认：…"）
  → 消息总线: clarification-response { choice: "Developer" }
  → subagent 恢复执行
```

pi-agent-core 的 `steer()` 是所有调研框架中唯一支持 mid-stream 消息注入的原语，天然适合路由 clarification 信号。

**并发模型**：所有 agent 共享 JS 事件循环。每个 agent 的 LLM 调用是 async（fetch/SSE）——Agent A 等待 LLM 响应时，Agent B 可以执行工具，主 agent 可以处理用户输入。协作式异步多任务，无需 Web Worker（agent 是 I/O 密集型，不是 CPU 密集型）。

**AgentOrchestrator**（~200-300 行 TypeScript，零新依赖）：

```
AgentOrchestrator
  ├── delegate(task) → 创建 subagent，后台执行，返回 subagentId
  ├── cancel(subagentId) → abort() + 消息通知
  ├── getStatus() → 所有任务状态列表（供 UI 渲染 badge + 任务列表）
  └── on(event, handler) → 订阅编排事件（供 Chat 面板监听完成/求助通知）
```

### ReAct 验证闭环

agent 行动后有 Observe 步骤（类似 Claude Code 跑 typecheck 验证）：

```
Think: 用户 ⌘K "帮我整理今天的笔记"，识别出 3 条会议记录
Act:   添加 #meeting 标签，填充 People/Topic 字段
Observe:
  "老王"在图谱中有对应节点吗? → 有"王海" (#colleague) → 链接
  对应 #skill 的修正历史? → People 字段从未被修正 → 模式可靠
```

**网页操作的 ReAct 闭环依赖 CDP 截图**（见第八节"浏览器权限"）：

```
Think: 用户 ⌘K "帮我提取这个页面所有模型的价格"
Act:   attach debugger → 点击 "View all models" 展开
Observe:
  CDP Page.captureScreenshot（后台，不切 tab，不抢焦点）
  → 截图确认：内容已展开，表格可见
Act:   提取表格数据 → 创建 #llm-model 节点
Observe:
  截图确认：数据完整（对比表格行数 vs 提取节点数）
  → 匹配 → detach debugger → 结果呈现在画板上
```

### 事件驱动路由（v6 修订）

大部分 AI 工作从"自动触发"变为"用户召唤触发"：

```
运行时路由（确定性，零 token 成本）:
  ClipAction       → 主 agent 匹配提取类 #skill
  TagCreated       → 主 agent 匹配字段建议 #skill
  FieldCreated     → 主 agent 匹配类型建议 #skill

模型路由（需要理解意图，消耗 token）:
  Chat 指令        → 主 agent → 理解意图 → 匹配 #skill → 执行（可能创建 subagent）
  /review 命令     → 主 agent → 匹配 Review #skill → 认知镜像（结构变化、矛盾、同构）

后台路由（LLM，低频后台执行）:
  Schema evolution skill 定期扫描 CRDT OpLog → LLM 分析修正模式 → 更新 #skill 节点规则
```

大部分 LLM 调用集中在 chat 指令、clip、/review 和后台 skill 迭代四个入口。前三个用户主动触发，成本可预测；后台 Schema evolution skill 低频扫描 OpLog + LLM 分析修正模式，成本较低但不为零。

### SKILL.md 兼容（远期）

soma 的 skill 节点与 Agent Skills 开放标准（agentskills.io）双向兼容：

- **导入**：SKILL.md → 解析为 #skill 节点 + rules 子节点 + tagDef。导入后 skill 即活——agent 开始使用它，从实践中积累新规则
- **导出**：#skill 节点 → SKILL.md。学到的新规则合并到 Rules 段落，失败记录变成 Known Issues

SKILL.md 是**分发格式**（静态、可共享），节点是**运行时格式**（动态、有状态、自进化）。当前阶段不需要 Pack/Plugin 分发层——skill 直接作为节点存在图谱里，等生态成熟后再加导入导出。

### 工具设计：3 个工具，action 参数分发

> **设计理念**：成功的 agent 系统收敛到极少数工具——Pi Agent 4 个、Codex CLI 2 个、Computer Use 3 个。原因：工具越少，模型选择越准确，KV-cache 效率越高，token 预算越省。soma 将所有 AI 能力浓缩为 3 个工具。
>
> 研究了 Claude Code、Manus、OpenClaw、Pi Agent、Claude in Chrome、Playwright MCP、browser-use 等 7 个系统的工具设计后得出此方案。

**Tool 1: `node` — 知识图谱操作**

```
node({
  action: "create" | "read" | "update" | "delete" | "search",

  // create
  name?: string,
  parent?: string,         // 节点 ID 或 "today"
  position?: number,
  tags?: string[],         // tag 名称，不存在时自动创建 tagDef
  fields?: Record<string, string>,  // field name → value

  // read（渐进式披露）
  id: string,
  // → 返回当前节点完整信息（name、tags、fields）
  //   + 直接子节点摘要（name、hasChildren、childCount）
  //   子节点的 tags/fields/content 不展开——需要则再 read

  // update（create 的所有字段 + 以下）
  id: string,
  remove_tags?: string[],
  remove_fields?: string[],
  move_to?: { parent: string, position?: number },

  // delete
  id: string,

  // search（临时查询，可保存为 Search Node）
  query?: string,          // 自然语言 / fuzzy（fuzzySort 名称匹配）
  tags?: string[],         // → HAS_TAG queryCondition
  fields?: Record<string, string>,  // → FIELD_IS queryCondition
  date_range?: { after?: string, before?: string },  // → CREATED_LAST_DAYS
  limit?: number,
})
```

**read 的渐进式披露**（参照 Claude Code 的 Read/Glob 模式）：

agent 不会一次获得整棵子树，而是逐级探索——在每一层做出知情决策，而不是被巨大的 tree dump 淹没：

```
read("node_abc")
→ {
    id: "node_abc",
    name: "和老王聊了搜索优先级",
    tags: ["#meeting"],
    fields: { People: "老王", Topic: "搜索优先级" },
    children: [
      { id: "child_1", name: "结论：搜索 > AI", hasChildren: false },
      { id: "child_2", name: "后续行动", hasChildren: true, childCount: 3 },
    ]
  }

// agent 决定 "后续行动" 值得深入 → 再 read
read("child_2") → 返回下一级...
```

这自然控制了 token 预算：agent 每次只加载它需要的一层。搜索返回更轻量的摘要列表（不含 children），agent 找到感兴趣的节点后再 read 深入。

**关键决策**：
- **read vs search 分离**：read 按 ID 读单个节点（高频，完整信息 + 子节点摘要），search 按条件查找（返回轻量列表）。返回格式完全不同，合为一个 action 反而更混乱
- **渐进式披露**：read 只返回当前节点 + 直接子节点摘要。agent 像人类一样逐级展开，避免一次性 dump 整棵树
- tag/field 操作合并到 create/update 中，不单独拆分。创建节点时直接 `tags: ["#meeting"]`，update 时 `remove_tags: ["#AI"]`
- Schema 操作（创建 tagDef/attrDef）通过 create + 特定 tag 实现，不需要独立的 Schema Tools

**search 的实现：复用现有 Search Node 基础设施**

agent 的 search 内部构造 queryCondition 树 + 调用现有的 `runSearch()` 引擎，**临时查询和持久化 Search Node 用同一套数据结构**：

```
agent search（临时查询）
  → 参数映射为 queryCondition 树:
      tags: ["#meeting"]         → HAS_TAG condition
      fields: { People: "老王" } → FIELD_IS condition
      date_range: { after: ... } → CREATED_LAST_DAYS condition
      query: "RAG"               → fuzzySort() 名称匹配（叠加在条件过滤之上）
  → 调用 evaluateCondition() + runSearch() 执行
  → 返回轻量摘要列表（Chat 中呈现）

用户: "这个搜索有用，保存一下" / 点击 📌
  → 同一棵 queryCondition 树 → createSearchNode() 持久化
  → refreshSearchResults() 物化 reference 节点
  → 画板上出现 Search Node（可手动排序、加视图）
```

**为什么这样设计**：
- agent 不需要替用户决定"这个查询值不值得保存"——临时查询零成本，用户自己判断
- "保存"不是重新构建，而是把已有的 queryCondition 树直接持久化——零浪费
- 符合渐进式交互：先轻量尝试 → 满意了再固定（和 Ghost UI 同一哲学）
- 代码层面完全复用 search-engine.ts 的 `evaluateCondition()` + `runSearch()`，不需要建新的查询引擎

**Tool 2: `browser` — 统一浏览器操作（16 action）**

> 研究了 Claude in Chrome 全部 18 个工具后重新设计。CiC 的工具拆分是历史包袱（`computer` 工具 baked in 模型训练，其他工具不能合入否则模型混淆）。soma 从零设计，单一工具 + action 参数是更自然的选择。

```
browser({
  action:
    // ── 观察（4）── 理解页面状态
    | "read_page"        // a11y 树 + ref IDs（结构化，用于规划交互）
    | "get_text"         // 纯文本提取（轻量，用于内容理解/clip）
    | "find"             // 自然语言找元素 → ref IDs
    | "screenshot"       // 截图（ReAct 验证 + clip 预览）

    // ── 交互（6）── 操作页面元素
    | "click"            // 点击（left/right/double + modifiers + hover）
    | "type"             // 文本输入
    | "key"              // 键盘快捷键
    | "scroll"           // 滚动（方向+量，或 ref 滚动到元素）
    | "drag"             // 拖拽（起点 → 终点）
    | "fill_form"        // 表单填充（ref + value）

    // ── 控制（3）── 页面/tab 管理
    | "navigate"         // URL 导航 / back / forward
    | "tab"              // list / create / switch
    | "wait"             // 等待页面渲染（SPA 必须）

    // ── 执行（1）── 页面上下文
    | "javascript"       // 执行 JS（SPA 数据提取）

    // ── 调试（2）── 浏览器内部状态
    | "read_network"     // 网络请求（API 数据提取）
    | "read_console",    // 控制台日志

  // ── 统一目标定位（三模式自动降级） ──
  target?: string,          // NL "the login button" / ref "ref_1" / CSS "#submit"
  coordinate?: [x, y],     // 像素兜底

  // ── read_page（渐进式披露） ──
  depth?: number,           // a11y 树深度（默认 15）
  filter?: "interactive" | "all",
  ref_id?: string,          // 聚焦到子树

  // ── find ──
  query?: string,           // "search bar", "price table"

  // ── screenshot ──
  fullpage?: boolean,
  region?: [x0, y0, x1, y1],  // 局部放大查看

  // ── click ──
  click_type?: "left" | "right" | "double",  // 默认 left
  modifiers?: string,       // "ctrl+shift", "cmd"
  hover_only?: boolean,     // true = 只悬停不点击

  // ── type ──
  text?: string,

  // ── key ──
  key?: string,             // "Enter", "cmd+a", "Backspace Backspace"
  repeat?: number,

  // ── scroll ──
  direction?: "up" | "down" | "left" | "right",
  amount?: number,
  // 也可用 target/ref 滚动到指定元素

  // ── drag ──
  start?: [x, y],          // 起点坐标（终点用 coordinate）

  // ── fill_form ──
  ref?: string,
  value?: string | boolean | number,

  // ── navigate ──
  url?: string,             // URL / "back" / "forward"

  // ── tab ──
  tab_action?: "list" | "create" | "switch",
  tab_id?: number,

  // ── wait ──
  duration?: number,        // 秒数（最大 30）
  wait_for?: string,        // CSS selector（元素出现后继续）

  // ── javascript ──
  code?: string,

  // ── read_network ──
  url_pattern?: string,
  limit?: number,

  // ── read_console ──
  pattern?: string,         // regex 过滤
  only_errors?: boolean,
})
```

**返回值**：
- `read_page` → a11y 树 + 交互元素 ref IDs（支持 depth/ref_id 渐进式披露）
- `get_text` → 纯文本（优先文章内容，轻量）
- `find` → 匹配元素列表 + ref IDs
- `screenshot` → base64 图片（支持 region 局部放大）
- `read_network` → 请求/响应列表（支持 url_pattern 过滤）
- `read_console` → 控制台消息列表（支持 pattern/only_errors 过滤）
- 交互 action → 操作结果 + 操作后页面状态摘要

**设计来源**：

| 特性 | 来源 | soma 实现 |
|------|------|----------|
| NL 元素查找 | CiC `find` | `find` action + `target` 参数 NL 模式 |
| ref ID 定位 | CiC `read_page` / Playwright | `read_page` 返回 ref，后续操作复用 |
| 坐标兜底 | Computer Use | `coordinate` 参数，ref 失败时使用 |
| a11y 树 + 渐进披露 | CiC `read_page` | depth/filter/ref_id 参数 |
| 纯文本提取 | CiC `get_page_text` | `get_text` 独立 action（token 成本远低于 read_page） |
| 结构化表单 | CiC `form_input` | `fill_form` action（ref + value） |
| 截图 + 局部放大 | CiC `computer.screenshot/zoom` | `screenshot` + `region` 参数合并 |
| trusted events | CiC `computer` | click/type/key/hover/drag 通过 CDP `Input` 域 |
| JS 执行 | CiC `javascript_tool` | `javascript` action |
| 网络请求 | CiC `read_network_requests` | `read_network` action |
| 控制台 | CiC `read_console_messages` | `read_console` action |
| tab 管理 | CiC `tabs_context/create` | `tab` action 合并 |
| 等待 | CiC `computer.wait` | `wait` action（duration + wait_for） |

**CDP 权限验证**：16 个 action 全部可通过 `debugger` 权限（Chrome DevTools Protocol）实现：

| action 分类 | CDP 域 | 必须 CDP？ | 说明 |
|---|---|---|---|
| **screenshot** | `Page.captureScreenshot` | ✅ 必须 | 唯一能后台截图不抢焦点的方式 |
| **click/type/key/hover/drag** | `Input.dispatch*Event` | ✅ 必须 | `isTrusted: true` 事件只有 CDP 能发 |
| **read_network** | `Network.getResponseBody` | ✅ 必须 | webRequest API 拿不到 response body |
| **read_console** | `Runtime.consoleAPICalled` | ✅ 必须 | 唯一能捕获控制台输出的途径 |
| **read_page** | `Accessibility.getFullAXTree` | CDP 更好 | Content Script 可遍历 DOM，但 CDP a11y 树更完整 |
| **find** | 基于 read_page + LLM | 同上 | 依赖 read_page 的质量 |
| **fill_form** | `DOM` + `Input` | CDP 更可靠 | 自定义下拉/日期选择器需要 trusted events |
| **javascript** | `Runtime.evaluate` | `scripting` API 可替代 | MAIN world 注入覆盖大部分场景 |
| **get_text** | — | ❌ 不需要 | Content Script `innerText` 够用 |
| **scroll** | — | ❌ 不需要 | Content Script `scrollTo()` 够用 |
| **navigate** | — | ❌ 不需要 | `chrome.tabs.update()` 即可 |
| **tab** | — | ❌ 不需要 | `chrome.tabs.*` API |
| **wait** | `Page.loadEventFired` | CDP 更精确 | setTimeout polling 也可用 |

> 16 个 action 中 6 个**必须** CDP、5 个 CDP 更好、5 个不需要 CDP。分层实现策略：Content Script 处理轻量操作（零警告栏），CDP 按需 attach 处理重活（attach 时出现警告栏，detach 后消失）。

**核心 ReAct 流程**：
```
read_page（渐进式：先 depth=3 扫描，再 ref_id 聚焦）
  → find（NL 定位目标元素 → ref ID）
  → click/type/fill_form（用 ref 交互）
  → wait（等待 SPA 渲染）
  → screenshot（验证结果）
  → 循环直到完成
```

**Tool 3: `undo` — 撤回操作**

```
undo({ steps?: number })  // 默认 1
```

撤回 agent 的上一步（或多步）操作。基于 Loro CRDT 的 `undoDoc()`，每个 agent commit 对应一次 undo。这是 agent 的安全网——做错了可以撤回，不需要用户介入。

**为什么只有 3 个工具**：

| 维度 | 15 个细分工具（旧方案） | 3 个工具（新方案） |
|------|----------------------|------------------|
| 模型选择准确率 | 低（15 选 1） | 高（3 选 1 + action） |
| System prompt token | ~3000 | ~800 |
| KV-cache 效率 | 差（长 tool schema） | 好（短 schema） |
| 维护成本 | 高（15 个 handler） | 低（3 个 handler + action 路由） |

### Reference 原则：AI 的每一个关联都应物化为连接

> **核心原则：AI 提到任何已有节点时，用 reference 而非纯文字。AI 发现节点间的关联时，将关联物化为 reference。**

Reference 是图谱变"厚"的方式。AI 加标签 = 分类（这是什么），AI 加 reference = 连接（这和什么有关）。标签是索引，reference 是脉络。

**两种 reference 形式**：

- **Inline reference**：嵌入文本流，自然阅读的一部分。如 "和 [[API 设计笔记]] 是同构"
- **Citation reference**：角标引用，不打断阅读，点击查看证据。如 "你开始用新框架思考问题¹²"

**在不同场景中的应用**：

| 场景 | 形式 | 创建图谱边？ | 示例 |
|------|------|:-----------:|------|
| AI 在画板上创建/修改节点 | inline ref 嵌入节点内容 | ✓ | Spark："和 [[API 设计笔记]] 是同构" |
| AI 在画板上组织节点 | reference node 作为子节点 | ✓ | 整理后加 `→ 搜索重构` 引用 |
| Chat 碰撞模式回答 | inline ref（可点击导航） | ✗ | "你写的 [[信息源管理]] 也在说这个" |
| Chat Review 输出 | citation 角标 | ✗ | "新结构浮现：干/湿框架¹²³" |
| Chat 执行报告 | inline ref | ✗ | "已给 [[老王聊天记录]] 加了 #meeting" |

画板上的 reference 是真实图谱边（持久化，参与检索和碰撞）。Chat 中的 reference 是导航链接（可点击跳转到 outliner 中的节点，但不创建图谱边——Chat 独立存储）。

**为什么这很重要**：

1. **强化检索策略**：我们不做 embedding，靠图结构检索。AI 积极创建 reference → 图更密 → agent 沿 reference 链发现关联 → 检索更准 → 碰撞更好。这是一个正向飞轮
2. **加速碰撞冷启动**：即使 Day 1-30 碰撞置信度不够展示在 Spark 中，AI 在 Chat 执行和 Clip 中已通过 reference 织出连接网。Month 2 碰撞启动时，这些 reference 让碰撞质量远高于纯 tag 匹配
3. **AI 回答可验证**：Chat 碰撞说"你之前写过 X"——如果是纯文字，用户要自己找；如果是 [[X]] reference，一点就到。可验证 = 可信赖
4. **Review 有据可查**：Review 的每个发现（新结构、矛盾、同构）附带 reference 到具体节点，从"AI 的观点"变为"有据可查的分析"

### 完整飞轮（v6）

```
用户记录（Node 输入，原样保留）
  ↓
用户召唤 AI（Chat 指令 / Clip / Tag 创建）
  ↓
主 agent 匹配 #skill → 执行（可能创建 subagent）
  ↓
ReAct 自检 → 结果呈现在画板上
  ↓
用户验证/修正（或直接编辑 #skill 节点）
  ↓
CRDT OpLog 天然记录修正信号
  ↓
Schema evolution skill 扫描 OpLog → 更新 #skill 节点规则
  ↓
下次同类任务 → 更准确

三条认知环（核心价值）：
  阅读环：浏览 → AI 处理结构 → 用户捕捉灵感 → 取景框改变
  碰撞环：初生想法 → Chat 加密度 → 新的问题和方向
  回顾环：积累 → Review 照亮认知演化 → 看见自己 → 改变下次阅读和思考

基础设施（润滑剂）：Schema evolution skill 让 AI 的「干」越来越准
```

## 十二、产品方法论修订（已确认）

> 以下修订在 v5+ 架构下已确认实施。v4 时期"AI 自动行动"的措辞已被"用户召唤时行动 + 静默 taste 优化 + 用户发起 Review"取代。

### 需要修改的

1. **"不做自动整理"** → **"AI 只在用户召唤时行动（chat/clip），在画板上可见可修正"**。Node 输入 AI 不碰；chat 指令和 clip 时 AI 在画板上行动，用户随时可修正。不透明的自动整理仍然不做。

2. **"剪藏"→"认知触发"** → clip 的目的不再是保存原文，而是提取思维结构 + 与已有笔记碰撞 + 邀请用户写自己的想法。原文不存储（Shadow Cache + TTL，AI 一次性完成理解），URL 即回溯路径。处理逻辑是进化的 #skill 节点，因人而异、因内容而异。

3. **Think Now 的表述** → 从"必须立即思考"调整为"选择是思考的第一步，深度思考可以渐进发生"。Think 是一个光谱：选择 → 标注 → 修正 → 连接 → 提炼。

4. **Connect 的定义扩展** → Connect 不只是"用户手动在笔记间发现脉络"，还包括"用户修正 AI 的组织"和"用户发起 Review 时发现自己的思考脉络"。修正和 Review 都是 Connect——用户在审视自己的内容和连接时，在做认知工作。

5. **去掉 Library/Inbox 作为系统容器** → 日期节点是唯一锚点。用户需要什么组织容器自己建节点即可。这消除了新用户需要学习的概念数量。

6. **Schema 由 AI 静默维护** → Schema 页面从"用户配置工具"变为"AI 工作的透明审查视图"。Schema evolution skill 从 CRDT OpLog 中分析修正模式（LLM 辅助），静默调整 schema 和 #skill 节点规则；用户在 chat/clip 结果中自然验证（不对就改），无需独立的审批环节。

### 不需要修改的

1. **"用自己的话写，才算真正理解"** — 核心信条不变。AI 降低到达这一步的门槛，但不替代。
2. **"AI 帮你发现和使用你的思考，而不是替你思考"** — 核心原则不变。AI 行动是"帮你组织"，不是"替你思考"。
3. **Think → Connect → Compound 框架** — 框架不变，但 AI 参与度提升，且 Connect 的含义扩展了。
4. **浏览器侧边栏定位** — 不变，且更强。浏览器是 Source 的天然入口，Outliner 是修正的天然界面。

## 十三、核心用户旅程（v6）

### Day 1：打开就是今天

用户装了 soma。没有 onboarding，没有"请先创建你的第一个笔记本"。

打开侧边栏，只有今天的日期。一个空的输入框。

下午看到一篇好文章，点 Clip。自动触发 Spark——AI 提取文章的思维框架（Day 1 的核心价值就是结构提取，碰撞尚无素材不强行展示）：

```
Mar 10, Monday
  └── RAG vs Fine-tuning: When to Use Which #article    ← #source 锚点
        ├── Highlights (field, 折叠)
        │     ├── "静态知识适合 fine-tuning，动态知识需要 RAG…"
        │     └── "选择取决于知识的时效性…"
        ├── #spark                                       ← AI 自动生成
        │     ├── 核心框架：知识时效性 → 技术选型
        │     │     └── → ref → "选择取决于知识的时效性…"
        │     └── （Day 1 无碰撞——置信度未达阈值，不硬碰）
        └── (URL / Author — field entries)
```

用户看到 Spark 的结构，被触发——写下自己的想法：

```
Mar 10, Monday
  └── RAG vs Fine-tuning: When to Use Which #article
        ├── Highlights (field, 折叠)
        ├── #spark
        │     └── 核心框架：知识时效性 → 技术选型
        ├── #note                                        ← 用户写的涟漪
        │     └── 也许我们应该先做 RAG——我们的数据几乎都是动态的
        └── (URL / Author)
```

**阅读环的第一次闭合**：AI 提取了文章的思维结构（取景框，不是摘要），Spark 作为持久化节点留在 #source 下，用户看到结构后被触发——写下的不是原文的结论，而是"这个取景框照到我的情况上是什么样"。#spark 和 #note 平级共存，互不干扰。原文已经退场，留下的是结构和用户的想法。**Day 1 的差异化就在这里**——用户第一次看到 AI 不是在帮他存文章或列要点，而是在帮他看穿文章背后的思考框架。

### Week 1：内容积累 + 第一次 Chat

15 条内容，全部挂在各天的日期节点下。用户随手写、Clip 文章，没有刻意整理。

周五下午，用户按 ⌘K 打开 Chat 面板：

```
用户: "帮我整理一下这周的笔记"

  ↓ 主 agent 直接在画板上操作

Agent: "整理好了！我做了这些：
       · 3 篇 AI 文章归到一起，加了 #article 标签
       · 4 条工作笔记归到一起
       · 2 条想法保留在原位
       画板上已经更新，你看看？"

用户: "工作笔记不要归一起，保持在各天下面"

Agent: "好的，已经移回去了。只保留标签分组。"
```

用户第一次看到 AI 在画板上工作——直接执行，在 Chat 里报告，用户给反馈就调整。不是"确认才做"，而是"做了不对就改"。

### Week 2：自建结构 + Tag 建议

用户第一次自己建了一个节点：`Projects`，把几条工作相关的笔记拖了进去。

```
Projects          ← 用户自建的，不是 soma 强加的
  ├── 搜索重构
  └── AI 策略探索
```

用户想给工作笔记加标签，创建了 `#meeting`。AI 建议字段：

```
创建 #meeting → AI 建议：
  ··People·· (引用)      ← Ghost field，点击接受
  ··Topic·· (文本)       ← Ghost field，点击接受
  ··Decision·· (文本)    ← Ghost field，忽略（用户不需要）
```

用户接受了前两个，忽略了第三个。Ghost UI 只在这个明确的"定义结构"时刻出现。

### Month 1：Chat 成为 AI 协作的日常

用户养成了习惯：日常在 outliner 里自由记录，需要 AI 帮忙时打开 Chat（⌘K）。

```
用户: "把这几条关于 RAG 的笔记总结一下"
Agent: "总结好了，放在今天的日期节点下。[查看]"

用户: "今天有个新任务：重构搜索模块，deadline 下周五"
Agent: "创建了 #task 节点，设置了 deadline。[查看]"

用户: "帮我从这个网页提取各家 LLM 的价格对比"
Agent: "提取中...（CDP 页面操作）已提取 5 个模型，放在新建的对比节点下。[查看]"
```

Chat 对话持久化，用户可以随时回顾之前的整理记录和上下文。

AI 的工作结果直接出现在画板上，和用户自己写的内容形式完全一样。

### Month 1.5：碰撞环首次出现 + 静默学习初见成效

用户注意到 clip 文章时 AI 不再乱标 #AI（之前改过 3 次，AI 静默学到了）。字段建议也更准了——不再出现 Decision 字段。**基础设施飞轮在运转——用户没有做任何配置，只是正常使用和修正。**

某天用户读到一篇关于"约束与创造力"的文章，突然想到一件事——打开 Chat，不是让 AI 做什么，而是想聊一个想法：

```
用户: "这篇文章说约束反而激发创造力。我刚想到，
       我们代码重构时限制接口反而设计更干净，
       这是不是一个更普遍的模式？"

Agent: "有意思。你上周写的信息源管理笔记里也提到了类似的东西——
       '关注 10 个公众号反而比关注 100 个看得更多'。
       这三个案例的同构是：下一层的约束消除了上一层的选择焦虑。

       不过你 3 月 5 日还写过'不要过早抽象'——
       这也是一种约束，但方向不同。两种约束各自适用的条件是什么？"

用户: "这个区分很重要。帮我把这个想法记下来。"
```

**碰撞环第一次闭合**：用户带着初生想法进入 Chat → AI 从用户已有笔记中找到碰撞素材 → 密度升高 → 产生了新的问题（"两种约束的区分"）→ 用户的取景框变了。

### Month 2：第一次 Review（认知镜像）

周末用户想回顾最近的思考，打开 Chat：

```
用户: /review this month

Agent: "几个认知结构层面的变化——

1. **新结构浮现**：你开始用'约束与自由'框架思考问题——
   它出现在代码重构、信息源管理、团队分工 3 个不同主题中，
   可能正在成为你的一个通用思考工具。

2. **旧结构升级**：你对 RAG 的理解从 Day 1 的直觉判断
   （'也许应该先做 RAG'）演化为互补策略
   （上周笔记：'不同场景用不同策略'）。你注意到这个转变了吗？

3. **未解决的张力**：'快速迭代'和'深度思考'在你的笔记中
   反复出现矛盾。这个张力还没有被你显式面对过。

4. **隐藏的同构**：你关于代码重构的笔记和关于信息源管理的笔记，
   结构上惊人相似——都在说'约束带来自由'。"

用户: "4 太有意思了，我完全没注意到。
       帮我把这两篇放在一起，我想写一篇关于'约束与自由'的思考。"
```

**回顾环第一次闭合**：Review 不是告诉用户"你做了什么"，而是让用户**看见自己看不见的东西**——新结构的浮现、未解决的矛盾、跨领域的同构。用户看见了自己的认知轮廓，产生了新的写作方向。

### Month 3：三条环协同运转

三条认知工作流开始形成协同：

- **阅读环成熟**：用户浏览网页时，侧边栏不再只是剪藏工具。AI 预处理结构，用户习惯性地写下"这让我想到什么"而非摘录原文。积累的想法改变了阅读视角——同一个主题的文章，现在看出了以前看不到的层次
- **碰撞环日常化**：用户养成了"有想法就打开 Chat 聊"的习惯。AI 能从 300+ 条笔记中找到碰撞素材，碰撞的密度和质量在提升
- **回顾环产出高价值内容**：`/review` 发现的同构和矛盾成为用户最有价值的笔记。"约束与自由"那篇从 Review 的发现开始，已经衍生出了一个持续生长的思考主题
- **基础设施稳定**：AI 的 #skill 节点趋于稳定——大部分偏好规则已从 CRDT OpLog 修正中迭代出来，三条环中 AI 的「干」越来越准

### Year 1：回头看

| | Day 1 | Month 1 | Month 3 | Year 1 |
|---|---|---|---|---|
| 内容量 | 3 | 60 | 300+ | 1000+ |
| 其中"自己的想法"占比 | 33% | 30% | 50%+ | 60%+ |
| Chat 碰撞模式频率 | 0 | 0 | 3/周 | 1/天 |
| Review 频率 | 0 | 0 | 1-2/月 | 随需 |

**Year 1 的核心变化**不是 soma 变聪明了多少，而是**用户的取景框变了多少**。那个 Day 1 随手写下"也许应该先做 RAG"的用户，一年后对同一个问题的理解深度完全不同——因为每个想法都经过了碰撞致密化，每次回顾都照亮了新的连接。soma 是这个过程的容器和镜子。

## 十四、杀手级场景：浏览器就是数据采集界面

### 问题

用户经常需要从多个网页采集结构化信息：对比各家 LLM 定价、从 LinkedIn 收集销售线索、调研竞品功能列表。传统工具的流程是：

```
在 Notion/Excel 定义表结构 → 打开网页 A → 手动复制粘贴 → 打开网页 B → 重复 → 发现缺列 → 回去改结构 → 重新填
```

用户在浏览器和笔记工具之间来回切换几十次。

### soma 的方式

soma 就在浏览器侧边栏里。AI 能读取当前网页。结构化数据模型能承载提取结果。三者结合：

**场景 A：AI 模型定价对比**

```
用户打开 Anthropic 定价页
侧边栏输入: "对比各家 LLM 价格"
  ↓ AI 读取当前网页，自动提取

对比各家 LLM 价格
  ├── Claude Sonnet 4
  │     #llm-model              ← schema 从第一个页面涌现
  │     Provider: Anthropic
  │     Input: $3 / 1M tokens
  │     Output: $15 / 1M tokens
  │     Context: 200K
  ├── Claude Haiku 4 ...
  └── Claude Opus 4 ...
```

用户切换到 OpenAI 定价页，在 Chat 中说"这个也加进去"：

```

  ├── ...Anthropic models...
  ├── GPT-4o                    ← 用已有 #llm-model schema 提取
  │     Provider: OpenAI
  │     Input: $2.50 / 1M tokens
  │     Output: $10 / 1M tokens
  │     Context: 128K
  └── GPT-4o mini ...
```

用户打开 Google 页面，发现 Gemini 有 Grounding 特性，手动加了 `Grounding: Yes`：

```
AI 学到: #llm-model schema 需要新增 Grounding 字段
  ↓ 自动回填已有的 Anthropic/OpenAI 模型: Grounding: No
```

**Schema 在使用中演化，AI 在用户修正后自动回填。**

**场景 B：LinkedIn 市场线索**

```
用户浏览 LinkedIn 目标公司 CTO 页面
侧边栏输入: "潜在客户"
  ↓ AI 读取 LinkedIn 页面

潜在客户
  └── Zhang Wei
        #contact
        Title: CTO
        Company: TechCorp →      ← 如果知识库已有则自动链接
        Location: Shanghai
        Source: linkedin.com/in/zhangwei
```

继续浏览下一个 LinkedIn 页面，一键添加。到第五个人时用户加了 `Company Size: 500+`，AI 尝试回填前四个人的公司规模。

### 为什么只有 soma 能做到

| 能力 | 为什么必要 | 谁有 |
|------|----------|------|
| 浏览器侧边栏 | AI 实时读取当前网页内容 | soma ✓ Sider ✓ Tana ✗ Notion ✗ |
| 结构化数据模型 | 提取结果是真正的 supertag/field，不是纯文本 | soma ✓ Tana ✓ Notion ✓ Sider ✗ |
| Schema 动态演化 | 用户加一个字段，AI 自动应用到同类节点并回填 | soma ✓ Tana ✗ Notion ✗ Sider ✗ |
| AI + 已有 schema 提取 | 第二个页面自动用第一个页面涌现的 schema | soma ✓ 其他全 ✗ |

**四项能力的交集只有 soma。** Tana 有结构但不在浏览器里。Sider 在浏览器里但没有结构化数据模型。Notion 两者都有一点但 schema 不能从行为中动态演化。

### 本质

> **浏览器就是数据采集界面，outliner 就是实时的结构化工作台。**

用户不需要"先定义表格结构再填数据"。他打开网页，数据就在侧边栏里结构化好了。Schema 从第一个页面涌现，在后续页面复用，在用户修正中演化。整个过程用户从未离开浏览器。

## 十五、技术路径与实施优先级

### 核心能力（按优先级，v6 修订）

1. **AI Gateway + BYOK** — 所有 AI 能力的前提。采用 pi-mono Proxy 架构：客户端 pi-agent-core（agent 循环）→ Cloudflare Worker pi-ai（统一 LLM API）→ AI 提供商。详见本节"AI 调用架构"

2. **主 agent + Chat 面板** — 独立 Chat 面板（⌘K 打开），对话独立持久化（不混入节点图谱），多轮上下文。主 agent 理解意图 → 匹配 #skill → 直接在画板上执行 → 在 Chat 中报告 → 接受用户反馈（同意/建议/撤回）。详见第十节"通道 2"。

3. **CDP 深度操作 #skill + 基础设施** — 通过 Chrome DevTools Protocol 深度操作网页：提取动态内容（点击展开、翻页滚动）、后台截图验证（不抢焦点）、表单填写（trusted events）。按需 attach/detach，用户召唤时激活。这是"浏览器就是数据采集界面"的基础。详见第八节"浏览器权限"。

4. **Clip 认知触发引擎** — 用户 clip 网页时，AI 提取内容的思维结构/取景框（不是摘要）+ 与用户已有笔记碰撞（找同构/冲突）+ 元数据填充（is/has/about）。处理逻辑是进化的 #skill 节点（不是固定模板），通过 OpLog 修正持续进化，因人而异、因内容而异。节点结构：#source（顶层锚点）→ #spark（AI 思维树，持久化）+ #note（用户笔记）平级共存，#highlight 作为 Highlights 字段默认折叠。触发策略：主动 clip = 自动 Spark，被动 clip（从 highlight/note 创建）= 手动 Spark 按钮，无 API key 不阻塞。详见第十节"通道 3"。

5. **Taste 学习（OpLog → #skill 节点）+ Review 引擎** — Schema evolution skill 读 CRDT OpLog，用 LLM 分析修正模式（原始信号零额外采集，分析需要 LLM），派生规则存为 #skill 子节点（可审查可修正）。Review 是独立的用户发起的内容反思工具（`/review`），照亮思考脉络和连接，对不同信息做**价值分层**（基于 #spark/#note 标签 + 连接密度 + 互动信号分配注意力权重）。两者是独立飞轮。

6. **上下文浮现** — 基于 #skill 节点中学到的用户偏好，在搜索时、写作时提供精准关联浮现。

### 实施优先级（MVP，v6 修订）

> **权限策略**：`debugger` 权限必须从 v0.1 就声明在 manifest 中。Chrome 扩展后补权限会自动禁用扩展，用户量越大流失越高。v0.1 就提供基础 CDP 功能（Clip 增强 + 智能截图），让权限从第一天就有合理用途。详见第八节"权限申请策略"。

```
P0 — 无需 LLM，纯本地 + CDP 基础（v0.1 上架）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
□ @引用 智能排序（最近使用 + 当前上下文相关度）
□ #标签 自动补全（已有标签模糊匹配）
□ 日期自然语言解析（"明天"、"下周一"、"3月15日"）
□ CDP Clip 增强: 动态页面内容提取（SPA、懒加载、展开折叠）
□ CDP 智能截图: clip 时自动截取页面可见区域作为预览

P1 — LLM + CDP 深度操作，用户召唤时触发（chat/clip/tag 创建）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
□ 主 agent: ⌘K 自然语言指令 → 在画板上执行
□ CDP 深度操作 #skill: 深度页面操作（翻页、多步提取、表单、ReAct 截图验证）
□ Clip 认知触发：LLM 提取结构 + 碰撞已有笔记 + is/has/about 元数据填充（进化 #skill）
□ 新建 supertag 时字段建议（Ghost fields）

P2 — Skill 迭代 + 环境智能
━━━━━━━━━━━━━━━━━━━━━━━━━
□ Schema evolution skill（扫描 CRDT OpLog + LLM 分析修正模式 → 更新 #skill 节点规则）
□ #skill 节点管理 UI（用户查看/编辑 AI 学到的规则）
□ Review 引擎（/review 命令 → 认知镜像：新结构、升级、矛盾、同构）

P3 — 高级自动化
━━━━━━━━━━━━━━━━━
□ Command Node（AI 工作流）
□ Event trigger（标签变更 → 自动填充字段）
□ 跨会话 taste 模型持久化
□ 语音输入 → 结构化笔记
```

### 两级模型调用策略

| Level | 运行位置 | 延迟 | 成本 | 适用 |
|-------|---------|------|------|------|
| L0 本地规则 | 浏览器 | <10ms | 零 | @引用、#标签、日期正则 |
| L1 LLM | API | 1-3s | 中 | Chat 指令执行、Clip 结构提取、Tag/Field 建议、/review、Schema evolution skill（OpLog 分析） |

**v6 关键变化**：L1 调用集中在 chat/clip/tag 创建/review 四个用户主动触发入口 + 后台 Schema evolution skill（低频）。不再有"每次 Enter 后 500ms debounce 触发 LLM"。不使用 embedding。
**缓存**：相同上下文的 LLM 结果缓存 1 小时（参照 Tana token caching 可降低 90% 成本）。

### 检索策略：图结构 + 渐进式披露，不做 embedding

**决策：不引入向量嵌入（embedding）。** 碰撞和 Review 的语义检索依靠节点图谱自身的结构——tags、fields、tree hierarchy——加上 AI agent 的渐进式发现能力。

**为什么不做 embedding**：
- RAG 准确率不够——向量相似度找到的是**主题相近**的内容（"都在讲 AI"），但碰撞最有价值的是**跨域同构**（"代码重构和信息源管理底下藏着同一个模式"），这需要结构理解而非表面语义匹配
- 额外基础设施成本高——需要维护向量索引、处理增量更新、选型 embedding 模型
- 节点图谱本身就是高质量索引——tags 提供分类维度，fields 提供结构化属性，tree hierarchy 提供上下文关系

**Agent 驱动的检索流程**（碰撞/Review 场景）：

```
AI 提取当前内容的核心结构/主题
  → node.search（按 tags、fields、date_range 查找候选节点）
  → node.read（渐进式披露——逐级读取候选节点内容）
  → LLM 判断语义关联强度（是否真正同构/冲突/互补）
  → 置信度过阈值 → 展示碰撞；未过 → 不展示
```

**与 embedding 方案的对比**：

| 维度 | Embedding + RAG | 图结构 + Agent 检索 |
|------|----------------|-------------------|
| 擅长发现 | 主题相似（表面语义） | 跨域同构（结构理解） |
| 每次碰撞成本 | 低（向量查询） | 中（多轮 LLM） |
| 基础设施 | 向量数据库 + embedding 模型 | 零额外基础设施 |
| 随图谱增长 | 需要增量更新索引 | 天然支持（search + read） |
| 准确率 | 中（召回多但噪声大） | 高（Agent 逐级判断） |

**成本控制**：agent 检索每次碰撞消耗更多 LLM token（多轮 search + read + 判断），但碰撞只在用户主动触发时发生（clip/chat/review），频率可控。加上 LLM 结果缓存和 Spark 碰撞的置信度阈值（低置信度不展示 = 不浪费 token），总成本可接受。

### AI 调用架构：pi-mono + Proxy 模式

> 评估了 [pi-mono](https://github.com/badlogic/pi-mono) 项目（`pi-ai` 统一 LLM API + `pi-agent-core` agent 循环），决定采用其核心抽象。

**选型决策**：

| 组件 | 选择 | 理由 |
|------|------|------|
| **统一 LLM API** | `pi-ai` | 20+ provider 适配（Anthropic/OpenAI/Google/...），MIT 协议，类型安全 |
| **Agent 循环** | `pi-agent-core` | ~570 行，零 Node.js 依赖，ReAct + steering + tool call，可在浏览器运行 |
| **运行位置** | Proxy 模式 | 客户端只需 fetch/SSE，服务端处理 provider SDK 差异 |

**Proxy 架构**：

```
┌─────────────────┐     fetch/SSE      ┌──────────────────────┐
│  Chrome 扩展     │  ←────────────→    │  Cloudflare Worker   │
│                 │                    │                      │
│  pi-agent-core  │                    │  pi-ai               │
│  (agent 循环)    │                    │  (统一 LLM API)       │
│  + tool handler │                    │  + provider SDK      │
│  + steering     │                    │  + API key 管理       │
└─────────────────┘                    └──────┬───────────────┘
                                              │
                                    ┌─────────┼─────────┐
                                    ↓         ↓         ↓
                               Anthropic   OpenAI    Google
```

**为什么 Proxy 而不是客户端直连**：
- Chrome 扩展不适合打包 20+ provider SDK（bundle size）
- API key 不应存在客户端（安全）
- Cloudflare Worker 天然适合做 API 代理（低延迟、全球边缘）
- BYOK 用户的 key 也通过 Worker 代理（加密传输，不在客户端明文存储）

**pi-agent-core 的 steering 机制**：`agent.steer(message)` 可以在 agent 执行中途注入用户反馈——跳过剩余 tool call，注入新消息。这直接映射到 soma 的"执行→报告→反馈"模式：

```
用户 ⌘K: "整理今天的笔记"
  → agent 开始执行（tool call: node.update × 5）
  → 执行到第 3 个时用户说 "#meeting 应该是 #1on1"
  → agent.steer("用户说 #meeting 应该是 #1on1")
  → 剩余 tool call 跳过，agent 重新规划
  → 用修正后的标签继续执行
```

### 浏览器能力分层

| 层级 | 权限 | 生命周期 | 用途 |
|------|------|---------|------|
| **L0 常驻** | Content Script + `activeTab` | 扩展安装后始终可用 | DOM 读取、基础 clip、高亮 |
| **L1 常驻** | `scripting` API (MAIN world) | 扩展安装后始终可用 | 访问页面 JS 上下文、SPA 数据 |
| **L2 按需** | `debugger` (CDP) | 用户召唤时 attach，完成后 detach | 深度页面操作、后台截图、trusted events |

L0/L1 在 manifest 中声明，用户安装时一次性授权。L2 按需激活——用户通过 chat 指令触发需要深度页面操作的任务时 attach，任务完成后 detach。**用户感知：召唤 AI 操作网页时出现提示栏，完成后消失。**

### MCP 的定位

106 说 MCP 必做。在 v6 的定位下，MCP 暴露节点图谱和 #skill 节点：
- 节点图谱：用户的全部内容 + AI 组织的结构（supertag/field/reference）
- #skill 节点：agent 的行为规则和从修正中学到的 taste 规则（供外部 AI 理解"这个用户怎么思考"）

## 十六、风险与失败模式

### 1. AI 行动的质量门槛

用户通过 chat/clip 召唤 AI 时，预期 AI 行动准确。如果结果离谱，用户会失去信任，不再召唤。比"没有 AI"更糟的是"AI 乱组织"。

**对策**：
- 渐进权限扩展（见第八节）：冷启动极保守，随 taste 积累逐步放开
- AI 没有行动时，系统退化为普通 outliner，仍然可用
- CRDT undo 作为安全网——即使 AI 做错了，修正成本极低
- 每个 AI 行动都有可见标记（区分 AI 做的 vs 用户做的），方便用户识别和修正
- **用户召唤 = 用户有预期**——v5 的优势是用户主动触发，心理上准备好了接受和修正结果

**更深层的认识**：知识组织不像代码编译——没有客观的对/错。AI 的"错误"分类可能恰好是用户没想到的视角。真正有害的不是"不完全准确"，而是"毫无道理"。只要 AI 的组织有其内在逻辑（即使和用户的第一反应不同），它就是一个有价值的起点——好过空白。

### 2. 修正信号的解读

用户对 AI 组织结果的反应只有三种，每种都是有效信号：

| 反应 | 含义 | taste 信号 |
|------|------|-----------|
| **没注意到** | 不在乎具体组织方式，主要通过 chat 消费知识 | 中性（组织方式对该用户不重要） |
| **看到了没改** | 觉得合理，或无关紧要 | 正向（OpLog 中的"未修正" = 默认接受） |
| **看到了改了** | 和 AI 的理解不同 | 最强信号（OpLog 中的修正模式） |

**对策**：
- Schema evolution skill 从 CRDT OpLog 区分三种情况：agent 操作后被修正（强信号）、agent 操作后未被修正（弱正向）、用户自主操作（creation 信号，不用于 taste）
- 需要多次一致的修正信号才调整，单次不过拟合
- 用户的自主组织行为（不涉及 AI 操作）视为 creation 信号，不纳入 taste 学习

### 3. 冷启动

AI-first 产品的冷启动比工具型更难——用户期待 AI 很聪明，但 soma 的 AI 需要积累才能聪明。碰撞飞轮需要笔记密度，前 30 天碰撞素材稀疏。

**核心策略：结构提取是 Day 1 差异化，碰撞渐进增长。**

```
Day 1:  Clip → Spark = 高质量结构提取（取景框，不是摘要）+ 留白
        Chat 碰撞模式 = 通用知识加密度（不依赖笔记积累，Day 1 即可用）
        /review today = 今天记录的节点之间找连接（用户主动要的，薄连接也不反感）

Week 2+: Spark 中碰撞零星出现（置信度过阈值才展示，不硬碰）
Month 2+: 碰撞质量提升，跨时间跨主题的同构/矛盾开始浮现
Month 6+: 碰撞成为 clip 的主要价值
```

**对策**：
- **Day 1 差异化**：结构提取（取景框）本身就与 AI 摘要工具不同——用户第一次看到 AI 不是在列要点，而是在帮他看穿文章背后的思考框架
- **碰撞是 pull 不是 push**：Spark 自动触发时碰撞有置信度阈值，低密度时不强行展示。碰撞的主要入口是用户主动发起的 Chat 碰撞模式和 `/review`——用户有预期时薄连接也不反感
- **Chat 碰撞模式不依赖积累**：用户带着想法进入 Chat，AI 用通用知识做多角度碰撞（找类比、反例、前人思考）。历史笔记让碰撞更 personal，但没有历史时依然有价值
- 官方预置 schema 模板（#article、#meeting 等）让 clip/chat 从第一天就有基础能力
- 主 agent 从通用能力开始（整理、总结、提取），随 OpLog 中修正模式积累静默进化个性化规则

### 4. Source 存储

AI 理解内容需要原文作为输入。但永久存储原文有成本和隐私问题。

**对策**：Shadow Cache（IndexedDB + TTL 自动过期）。AI 在内容进入时一次性完成理解，理解结果（#spark 节点树 + is/has/about 元数据）永久存储为节点，原文可过期。

### 5. 过度结构化（Over-structuring）

Fabric 的"Death to Organizing"反而让用户焦虑——AI 的结构化替代了用户的心智模型。Mem 1.0 的全自动组织被批评为"不可信赖"。

**v5 已大幅降低此风险**——Node 输入 AI 不碰，只有 chat/clip/tag 创建时 AI 才介入。但仍需注意：
- 主 agent 执行结果不要过度结构化——用户说"整理一下"不等于"每条都加 5 个标签"
- Clip 提取结构 + 碰撞，不发明新 supertag，不把主题当标签，元数据按 is/has/about 填充
- 允许节点没有标签、没有字段——不追求 100% 分类
- **v5 的核心保障**：用户不召唤，AI 不出现。过度结构化的前提——"AI 在每次输入时自动行动"——已被移除

### 6. AI 的"恐怖谷"（Uncanny Valley）

MIT Technology Review 研究指出：AI 输出"几乎对但不完全对"比"明显错误"更让人恼怒——识别和纠正微妙错误需要更高认知负担。关键指标：**不确定性比错误更有害**——用户可以适应"AI 在 X 场景下总是犯错"，但无法适应"有时对有时错"。

**对策**：
- v1 只做高准确率场景（clip 提取、tag 字段建议），不做低准确率场景（内容改写）
- Ghost UI 视觉强度与置信度正相关——高置信度实线，低置信度虚线
- 给用户稳定的心智模型——"Chat 指令 AI 执行，Clip 时 AI 提取结构+碰撞，其他时候 AI 不出现"

### 7. 产品叙事

**对外叙事保持简单**：

> **Think where you read. The longer you use it, the clearer you see.**

用户感知到的不是"工具变聪明了"，而是"我看得更清楚了"。三条认知环在幕后运转——阅读时 AI 处理结构让用户专注灵感，Chat 中碰撞让想法致密化，Review 照亮自己的认知演化。用户不需要知道这些机制，只需要感受到：读得越多想得越深，想得越深看见的越多。

```
多 Agent 通信     → 10+ 框架调研（Google ADK / OpenAI / LangGraph / Vercel AI SDK / Claude Code 等）→ 无现成方案 → 自建 AgentOrchestrator：Loro CRDT blackboard + EventTarget 消息总线 + pi-agent-core steer() 求助流
冷启动与商业模式  → 检索策略确认（图结构 + 渐进式披露，不做 embedding）→ 碰撞是 pull 不是 push（Spark 置信度阈值 + Chat/Review 用户主动发起）→ Day 1 差异化 = 结构提取质量 → 商业模式 = BYOK/OAuth 先行 + Outliner 收费
Reference 原则    → AI 提到节点用 reference 不用纯文字，关联物化为图谱边 → 画板 = 真实 reference（inline ref / reference node），Chat = 导航链接（inline ref / citation 角标）→ 强化图结构检索 + 加速碰撞冷启动 + AI 回答可验证
```

### 定位

> **soma 是一个安静的知识环境，也是一面认知的镜子。打开就是今天，想到什么写什么，看到什么 clip 一下。Outliner 是用户和 AI 共同持有的画板——用户在上面自由记录，AI 在用户需要时通过 chat 指令在同一块画板上行动。三条认知工作流——阅读时捕捉灵感、Chat 中碰撞想法、Review 时照亮认知演化——让用户在使用中越看越清楚。AI 的静默学习是让这个循环越转越顺的基础设施。**

### 它为什么独特

1. **零概念启动**（vs 所有竞品）——没有 Library/Inbox/Notebook 需要学习，打开就是今天，写就行
2. **输入即保留**（vs v4 的"自动结构化"）——用人话写，原样保存，outliner 保持快速和纯净
3. **三条认知环**（vs 所有竞品只做"工具优化"）——阅读时捕捉灵感、Chat 中碰撞想法、Review 时照亮认知演化。环闭合的是用户的认知，不是软件的模型
4. **Chat 双模式**（vs Fabric/Mem 的全自动）——执行模式（帮我整理）+ 碰撞模式（我刚想到一个东西），AI 既是执行者也是思维碰撞伙伴
5. **共享画板**（vs Fabric/Mem 的黑箱）——outliner 是用户和 AI 的共享工作面，谁画的都可见可修正
6. **Schema 由 AI 维护**（vs Tana/Notion）——用户不需要理解 supertag/field/data type
7. **静默学习**（基础设施）——AI 在幕后从修正模式中迭代规则，让三条环中 AI 的「干」越来越准
8. **浏览器原生**——Source 的天然入口，画板的天然界面
9. **安静**——AI 安静融入工作流，非必要时完全静默，等待用户召唤
