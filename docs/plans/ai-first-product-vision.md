# soma Product Vision v2

> Think with your AI.

## 一、产品形态

一个意图入口。当前主形态是一个输入框：用户想到什么就写什么，记录、提问、指令、随想都从这里进入。AI 理解意图，做出最合适的响应。

```
用户：今天会议决定 Q3 涨价 15%

soma：已记录。
     → 你之前有一条笔记"竞品定价对比"提到竞品 Q2 涨了 10%，
       这次涨幅比竞品更激进，要关注一下吗？
```

```
用户：帮我整理上周的会议笔记

soma：找到 4 条会议笔记，按主题整理如下：
     [内嵌节点列表，可展开、可编辑、可跳转]
```

```
用户：如何看待 AI agent 的发展趋势

soma：基于你之前的 3 条相关笔记和当前页面...
```

**用户不需要区分"记笔记"还是"跟 AI 聊天"。** 写进去，得到有价值的回应。AI 的回应自然地包含记录、连接、分析。

背后有一整个知识图谱（节点树）作为 AI 的记忆。用户偶尔需要时，可以打开 Outliner 直接查看和编辑节点，但这不是默认动作，而是深入操作时的选择。

## 二、核心理念

### 从 v1 到 v2 的演进

v1 围绕 **Think → Connect → Emerge** 三步循环（见 `product-philosophy.md`）。每一步依赖用户主动操作：手动记录 → 手动组织 → 等待涌现。

v2 的核心转变：

| v1 | v2 |
|----|-----|
| Think where you read | Think with your AI |
| 工具 + AI 辅助 | 人机协作 |
| 用户是数据主人，AI 是工具 | 用户和 AI 都在同一块画板上行动 |
| Think → Connect → Emerge 串行循环 | 对话中三者一体 |
| 多个功能入口 | 一个意图入口 |
| Outliner 是主界面 | 意图入口是主界面，Outliner 是可选的深入操作层 |

### 不变的理念

- **工具应该让人变强，而不是让人觉得工具很强** — AI 协作的目标是用户获得新理解
- **知识组织没有客观正确答案** — AI 的"错误"组织可能是惊喜
- **正反馈是成功的关键** — 对话本身就是即时反馈

## 三、人机协作

### AI 是协作伙伴，不是工具

用户和 AI 在同一块画板（节点树）上共同工作。任何一方都可以创建、编辑、连接节点。

- **用户输入** → AI 理解意图，执行最合适的动作（记录 / 搜索 / 分析 / 操作）
- **AI 主动行动** → 发现重要内容时创建节点、发现连接时建立关联
- **一切可审查、可修正、可删除** — 协作不等于失控

### 区分来源，不区分行动权

AI 创建的内容标记来源（如小标识），用户可以确认、编辑、删除。编辑过的变成共创内容。节点树从"用户的笔记本"变成**人机共同的思考痕迹**。

### 对话即思考

不再是先记录再组织再等待涌现，对话中三者同时发生：

- 用户带着想法来对话 → **记录**（Think）
- AI 拉入相关笔记，发现连接 → **组织**（Connect）
- 用户看到意想不到的关联 → **涌现**（Emerge）
- 有价值的部分自动沉淀为节点 → 下次对话时被 AI 拉入

**飞轮：对话 → 沉淀 → 对话 → 沉淀。** 每次对话都在丰富共享画板，画板又让下次对话更深入。

## 四、认知摩擦

### 为什么需要摩擦

AI 协作中最省力的路径是"让 AI 做完，我看结果"。但这让用户变成消费者，没有思考，也没有成长。

soma 的独特定位：**在正确的地方主动增加认知摩擦。** 不是阻碍用户，而是邀请用户做一个判断。判断 = 思考 = 成长。

**摩擦不是为了确认操作，而是为了放大思考。**

### 摩擦的触发原则

> **摩擦带来的收获 > 用户被打断的成本，才触发。**

不是有机会就触发。大部分时候 AI 应该安静地执行，记录、整理、关联、沉淀都可以直接推进。只有当 AI 发现了**值得用户判断的认知信号**时，才递出一个思考机会。

值得触发摩擦的信号包括：
- 这和你之前的观点可能矛盾
- 你最近反复回到同一个主题
- 两个原本分离的主题可能有交集
- 当前结论背后缺一个关键证据
- 你已经积累到足以提炼出更高层概念

好的摩擦：
- "这跟你上周记的 X 好像矛盾？" → 用户发现了自己观点的变化（高收获）
- "你最近三次讨论都提到了 Y，值得提炼成一个主题吗？" → 用户看见了自己的模式（高收获）

坏的摩擦：
- 每次记录都问"要不要加标签？" → 打断流畅的记录（高成本，低收获）
- 每次搜索都问"哪些最相关？" → 用户已经在看了，不需要被问（无意义）

**形式是思考入口，不是审批流程。** 它通常应该是一个轻量提示、一个选择题、一个建议分支，而不是 modal、确认框或连续 yes/no 问题。主路径始终顺滑，摩擦只在值得的时候点亮。

### 摩擦的度

| 太少 | 恰到好处 | 太多 |
|------|---------|------|
| AI 做完一切，用户只看 | AI 先推进，在高价值时刻邀请判断 | 每次操作都要确认 |
| 用户没有成长 | 用户在判断中获得新理解 | 用户烦了就不用了 |
| ≈ 纯 ChatGPT | ≈ soma | ≈ Clippy |

## 五、真实使用场景

### 早上 — 打开 soma

用户点击 soma 图标，侧边栏打开。首屏**不依赖 AI 调用**，本地数据即时渲染：

```
┌──────────────────────────────┐
│  soma                    [⚙] │
│                              │
│  Recent                      │
│  • Q3 涨价决策          昨天  │
│  • 竞品定价对比         昨天  │
│  • AI Agent 调研        昨天  │
│                              │
│  ───────────────────────     │
│  Last chat                   │
│  "帮我整理定价相关笔记"       │
│  ───────────────────────     │
│                              │
│  ┌──────────────────────┐    │
│  │ 写点什么...       [@] │    │
│  └──────────────────────┘    │
└──────────────────────────────┘
```

最近的节点 + 最近的对话 + 输入框。用户想浏览就浏览，想聊就聊。打开即可用，无需等待。

### 浏览文章 — 静默输入 + 对话

**不打开侧边栏**：选中一段文字 → 高亮工具栏 → 点高亮 → 存节点 → 继续阅读。完全静默。

**打开侧边栏**：

```
用户：这篇文章的核心观点是什么

soma：这篇文章对比了 5 个 AI Agent 平台...

     跟你之前的 @AI Agent 服务商调研 有交集——
     那条笔记里你已经分析了 Okara 和 CrewAI，
     这篇新增了 AutoGen 和 LangGraph 的对比。
     要合并到一起吗？

用户：好的，合并

soma：已合并到 "AI Agent 服务商调研"：
     ┌─ AI Agent 服务商调研 ────────┐
     │  • Okara.ai — SEO Agent     │
     │  • CrewAI — 多 Agent 框架    │
     │  • AutoGen — 微软开源        │  ← 新增
     │  • LangGraph — LangChain    │  ← 新增
     │  ...                 [打开 ↗] │
     └────────────────────────────┘
```

用户点 [打开 ↗] → Outliner 面板打开，可直接编辑。

### 快速记录 — AI 安静地记

```
用户：定价策略的核心不是价格本身，而是用户对价值的感知

soma：已记录。
```

大部分时候 AI 安静确认。只有发现高价值关联时才补充：

```
soma：已记录。
     → 这跟你上周高亮的一段话呼应——
       "Stripe 的定价页不展示价格，展示的是每个方案能帮你做什么"
```

### 准备工作 — AI 整理 + 内嵌节点

```
用户：帮我整理所有关于 Q3 定价的笔记，下午要汇报

soma：找到 6 条相关笔记，整理为：

     ┌─ Q3 定价汇报准备 ──────────┐
     │  决策                       │
     │  • 涨价 15%（3/18 决定）    │
     │  • 生效时间：Q3 第一天      │
     │                             │
     │  依据                       │
     │  • 竞品 Q2 涨了 10%        │
     │  • 用户调研：价格敏感度低    │
     │                             │
     │  风险                       │
     │  • 企业客户可能比价         │
     │  ...              [打开 ↗]  │
     └────────────────────────────┘

     按"决策/依据/风险"做了分组。
     还需要加什么维度？
```

### 回顾 — 对话中自然涌现

```
用户：这周我都在想什么

soma：这周你的笔记集中在两个主题：

     1. 定价策略（8 条）— 从竞品调研到涨价决策，
        形成了完整的决策链

     2. AI Agent 生态（5 条）— 从服务商调研到
        架构思考，但还没有形成结论

     有意思的是，这两个主题可能有交集——
     你记了一条"Agent 服务商的定价模式值得研究"，
     但还没有展开。要深入聊聊吗？
```

v4 的 Emerge 不再需要用户主动 `/review`，在对话中自然发生。

## 六、产品架构

### 蓝图总览

从产品蓝图看，soma 是：

- **一个意图入口** — 用户写什么都行，系统理解意图
- **一块共享画板** — 用户和 AI 共同沉淀、编辑、连接节点
- **一个会推进的 AI 协作者** — 默认推进整理、连接、提炼
- **一套放大思考的认知摩擦机制** — 在高价值时刻递出判断机会
- **一个可回到结构层审查与精修的 Outliner** — 深入操作时接管
- **一个未来可承载任务、自动化和多 Agent 协作的长期系统**

### 一个意图入口 + 静默输入 + 共享画板

```
输入渠道:
  1. Chat 对话（主入口，侧边栏打开时）
     当前主形态是一个输入框，记录、提问、指令、随想都从这里进入
  2. 网页高亮 / 笔记 / 剪藏（静默输入，侧边栏无需打开）
     在网页上直接操作，零摩擦，无需 AI 参与

共享画板:
  LoroDoc 节点树 — 两个渠道的内容都沉淀在这里

输出界面:
  Chat — 所有价值主要在对话中兑现
  Outliner — 可选的深入操作与审查层
```

### 两种输入渠道

**Chat 对话**（主入口）：用户打开侧边栏，看到对话界面。写什么都行，记录、提问、指令、随想都可以。AI 理解意图，做出最合适的回应。

**静默输入**（辅助入口）：高亮、笔记、剪藏是在网页上的空间操作，通过 Content Script 完成，**侧边栏无需打开**。没有 AI 参与，没有对话回应，纯粹的零摩擦输入。

静默输入的价值不在操作时兑现，而在**之后的对话中兑现**：
- 用户高亮了一段关于定价的文字（静默，无打扰）
- 三天后用户在 Chat 中讨论定价策略 → AI 自然拉入这条高亮
- 用户说"最近我看了什么关于 X 的" → AI 能答出来

**静默输入丰富 AI 的记忆，对话是记忆兑现价值的出口。**

### 记忆分层：Chat 过程层 + Outliner 长期记忆

soma 里不应该出现一个用户不可见的独立 memory 产品层。更合理的分层是：

- **Chat transcript** — 协作过程层。保存"我们怎么走到这里"：来回讨论、判断过程、未完成的执行脉络。它独立于节点图谱长期存在，但不是知识的最终落点
- **Outliner / 节点树** — 长期记忆层。凡是用户应该能看到、修正、继承、引用的记忆，都应该是节点：长期偏好、持续主题、未完问题、工作方式约束、任务状态、AI 学到的规则
- **Runtime retrieval / cache** — 运行时辅助层。检索索引、相关性排序、摘要缓存、最近活动痕迹可以独立存在，但它们不是用户意义上的"记忆"，也不能成为真相源

判断标准很简单：**凡是用户可感知、可审查、可修正的长期记忆，都必须落为节点。**

这意味着未来的跨会话能力不是"让模型自动带着所有旧聊天"，而是两步：

1. **检索过去的过程历史** — 按时间找最近聊过什么，或按主题找过去怎么想过
2. **把真正稳定的东西沉淀为节点** — 当某个偏好、约束、主题或问题跨越多次对话仍然成立时，应该进入 Outliner，而不是永远埋在 transcript 里

### Chat 输入框增强：@ 引用节点

输入框支持 `@` 触发节点选择器，让用户精确地围绕某些节点聊天：

```
用户：帮我对比 @竞品定价对比 和 @Q3涨价决策 的矛盾点
用户：基于 @AI Agent 架构笔记 扩展关于多 Agent 编排的思路
用户：@今天的会议记录 提取 action items
```

输入 `@` → 弹出节点搜索下拉 → 选中后插入引用 → 节点内容作为精确上下文注入。类似 GitHub `@mention` 或 Slack `@user`，在原生 textarea 上实现（浮动下拉 + 引用标记），不需要富文本编辑器。

### 回应中的内嵌节点

AI 的回答不只是文本，而是可以嵌入活的节点。渐进式交互：

1. **默认态** — 标题 + bullet，和文本自然混排
2. **点击** — 展开 children（mini OutlinerView）
3. **按钮** — 跳到独立 Outliner 面板

嵌入的是真实节点引用，不是快照。Chat 和 Outliner 看到的永远是同一份数据。

### 浏览器上下文

soma 活在浏览器侧边栏，天然知道用户正在看什么。这个上下文自动注入对话，不需要用户解释。

- 用户在看一篇文章，直接问"这篇文章的要点" → AI 已经知道是哪篇
- 用户在看一个产品页，说"记一下这个定价方案" → AI 知道具体内容
- 用户在看自己之前的笔记，问"帮我扩展一下这个想法" → AI 知道上下文

### 预设召唤

用户主动设置持续运行的指令：
- "每天早上总结昨天的笔记"
- "浏览 AI 相关文章时提醒我相关笔记"
- "每周回顾 #research 下的新增内容"

不是 AI 自作主张推送，而是用户授权的自动化。

## 七、与 v1 产品哲学的关系

### 保留

- 核心价值观（让人变强、正反馈、少即是多）
- 浏览器侧边栏的位置优势
- "一切皆节点" 的数据模型
- 高亮 / 剪藏的零成本输入
- Supertag + Field 的结构化能力
- Spark 认知触发

### 演进

- **Think → Connect → Emerge** 不再是显式的三步循环，而是在对话中自然融合
- **"AI 只在用户召唤时行动"** 演进为 AI 可在协作中主动行动（标记来源，用户可审查）
- **多功能入口** 收敛为一个意图入口 + 浏览器空间操作（高亮 / 剪藏）
- **Outliner 作为主界面** 演进为 Outliner 作为可选的深入操作层

### 不变的设计红线

1. **用户永远可以覆盖 AI 的任何决定**
2. **AI 创建的内容标记来源**
3. **不是每次交互都要求用户判断，但高价值时刻要创造判断机会**
4. **笔记不够多时坦诚说"再积累一些"**
5. **预设召唤需要用户主动设置**
6. **用户可感知的长期记忆必须是节点，不放进黑箱 memory 数据库**

## 八、产品理念演进历程

### v1：记下来，用得上

> Slogan: "记下来，用得上"

- 方法论：**捕获 → 消化 → 重遇**（三步线性流）
- 核心假设：笔记是对信息的预处理
- AI 角色：未明确
- 主界面：Outliner

### v2：Think where you read

> Slogan: "Think where you read"

- 方法论：**Think → Connect → Compound**
- 关键变化：从"记录"升级为"思考"，"重遇"变成"复利（Compound）"
- 新增：费曼学习法理念，"用自己的话写才算理解"
- AI 角色：辅助

### v3：AI 策略对齐

> Slogan: "Think where you read"

- 方法论：**Think → Connect → Compound**（不变）
- 关键变化："选择是思考的第一步"（高亮即思考）、三条认知工作流（阅读环 / 碰撞环 / 回顾环）
- 新增：开放问题区（剪藏边界、上下文感知、AI 工作流决策）

### v4：AI 照亮你的思考

> Slogan: "Think where you read"

- 方法论：**Think → Connect → Emerge**（Compound → Emerge，强调涌现）
- 关键变化：AI 独立成章（"照亮你的思考"）、干 / 湿分工、Outliner = 共享画板、Taste 学习、设计红线
- 新增：市场定位图、产品路径（4 条）、"知识组织没有客观正确答案"
- 文档：`product-philosophy.md`（当前 v4 版本）

### v5：人机协作（当前方向）

> Slogan: "Think with your AI"

- 方法论：**对话即思考**（Think / Connect / Emerge 在对话中一体化）
- 关键变化：
  - 一个意图入口替代多入口
  - AI 从工具变为协作伙伴
  - 认知摩擦主动设计，用于放大思考而非确认操作
  - Outliner 从主界面变为可选深入操作层
  - 记录和对话的边界消失
- 文档：`ai-first-product-vision.md`（本文件）

### 演进脉络

```
v1  记录工具    "记下来，用得上"           捕获 → 消化 → 重遇
v2  思考工具    "Think where you read"    Think → Connect → Compound
v3  +AI 策略    "Think where you read"    + 认知工作流
v4  AI 照亮     "Think where you read"    Think → Connect → Emerge
v5  人机协作    "Think with your AI"      对话即思考
```

从"记录"到"思考"到"协作"，每次迭代都在把更多的认知工作从用户独自承担，转移到人机共同完成。核心不变的是：**工具让人变强，不是让人觉得工具强。**

## 九、异步任务与多 Agent 协作

### 任务即节点

不需要独立的任务管理系统。节点树本身就是任务系统：

```
"调研 AI Agent 竞品"  #task  status: running
  ├── 搜索主流平台          #task-step  status: done
  ├── 阅读 CrewAI           #task-step  status: running  ← subagent 1
  ├── 阅读 AutoGen          #task-step  status: running  ← subagent 2
  ├── 阅读 LangGraph        #task-step  status: pending
  └── 整理对比表            #task-step  status: pending
```

- `#task` 是 supertag，定义 `status` 字段（pending / running / done / failed）
- 任何节点打上 `#task` 就自动拥有状态
- 状态变更 = `node_edit` 修改字段值，subagent 用现有工具即可操作
- 任务完成的结果 = 子节点，天然在知识图谱中可搜索、可关联

这是 Spark 三态节点（pending → loading → complete）的泛化。当前 BulletChevron 的 spinner 逻辑可以从"检测 isSparkNode"泛化为"检测 status === running"。

### Chat 中的任务感知

节点树是数据层，但 Chat 界面需要任务感知视图：
- 当前有哪些 agent 在执行什么任务
- 每个任务的实时进度（checklist 自动更新，因为底层是节点字段变化）
- 已完成任务的结果（内嵌节点展示）
- 侧边栏关闭时，通过浏览器通知推送完成结果

### 多 Agent 协作网络

不是简单的"主 agent 派 subagent"单向指挥，而是多个 agent 的协作：
- Agent 之间可以相互询问信息
- Agent 之间可以通知状态变化
- 编排层（Orchestrator）协调任务分配和依赖关系

与 TASKS.md 中 **Layer 3: 编排**（AgentOrchestrator + subagent delegation + 后台任务 UI）直接相关，待后续深入设计。

## 十、2026-03-19 产品焦点收敛

### 产品形态确认

**Chat 是主界面，Outliner 是可选的查看/编辑层。**

用户（创始人自身）的真实使用方式：通过与 AI 对话进行思考，不喜欢手动组织，希望 AI 将讨论直接生成或更新为持久化内容。Outliner 在偶尔想深入思考或查看文档时使用。

| 层 | 角色 | 投入优先级 |
|---|---|---|
| **Chat** | 思考发生的地方，主交互面 | 最高 |
| **Node tools** | AI 操作知识图谱的手脚 | 最高 |
| **Outliner** | 查看产物、偶尔手动编辑 | 维护现状，不再大量投入 |

**这意味着**：停止打磨 outliner（选择、拖拽、键盘快捷键够用即可）。Outliner 代码保留，数据模型（节点/标签/字段）、CRDT 同步是核心资产——只是消费者从"用户直接操作"变成"AI 通过工具操作"。

### 工具层评估

现有 7 个 AI 工具（node_create / node_read / node_edit / node_delete / node_search / undo / browser）**对 v5 已经够用**。AI 能创建、读取、编辑、搜索、删除节点，对话沉淀通过现有 node_create / node_edit 实现，不需要额外工具。

**核心缺口不在工具层，在记忆层**：

| 缺口 | 说明 | 解法 |
|------|------|------|
| 跨会话记忆 | AI 无法回忆上次聊了什么 | Chat 持久化 + `past_chats` 检索工具（纯 chat 层，与 node 无关） |
| 上下文感知不足 | AI 不知道最近访问/编辑了哪些节点 | 丰富 system reminder 注入 |
| 主动沉淀行为 | AI 不知道什么时候该主动记 | system prompt 行为指导（简短即可） |

### 沉淀生命周期

AI 沉淀的内容不是"记了就完"，需要生命周期管理，防止 outliner 膨胀到不可用：

```
对话中产生 → 沉淀为 #memo 节点 → 定期整理 → 提炼/合并/淘汰
```

**`#memo` supertag** 标记 AI 沉淀的内容，附带结构化字段：
- **source**：来自哪次对话（chat session id）
- **status**：`fresh` → `reviewed` → `absorbed` / `obsolete`
- **category**：`decision` / `preference` / `insight` / `todo`

**整理机制**：`fresh` 的 memo 积累到一定数量或时间后，AI 审视这批 memo，发现模式、合并重复、淘汰过时的，把有价值的提炼为更持久的节点。整理完的 memo 标记为 `absorbed`（已吸收到其他节点）或 `obsolete`（已过时）。

**保守原则**：AI 智商够高，只需简短指导。只沉淀决策、结论、长期偏好这类跨会话仍然成立的内容。日常对话细节不沉淀。宁可少记，不要膨胀 outliner。

这是 v5 飞轮的实体化：**对话 → 沉淀 → 整理 → 更深的发现 → 下次对话更深入**。

### 下一步优先级

1. **Chat 持久化 + past_chats 检索**（ai-context-management Phase 1 + 新搜索工具）
2. **system reminder 注入最近活动**（丰富 AI 的上下文感知）
3. **system prompt 加入沉淀行为指导**（低成本）
4. **`#memo` supertag 定义 + 整理机制**（沉淀基础设施）

## 十一、未决问题

### 已拍板

- **AI 发现值得长期保留的内容时，可以直接记录到 nodes 中**。默认不是候选箱，也不是先请求确认；AI 作为协作者可以先动手，用户再修正
- **past chats 的用户心智更接近全局个人历史，而不是当前局部空间**
- **过去对话 / 长期记忆的来源默认收起，只在展开时显示**。引用走渐进式披露，不做默认重 citation UI
- **durable memory 必须是结构化节点数据**，不做黑箱 memory 条目
- **产品焦点：Chat 是主界面，Outliner 是可选查看/编辑层**（2026-03-19 焦点收敛）
- **工具层够用，核心缺口在记忆层**：跨会话记忆（past_chats 检索）+ 上下文感知 + 沉淀行为指导
- **沉淀需要生命周期管理**：`#memo` supertag + status 流转 + 定期整理提炼，防止 outliner 膨胀

### P0：必须先回答的问题

1. **什么东西应该从 Chat 沉淀成节点**
   现在已经明确 durable memory 属于 Outliner，且 AI 可以直接记进 nodes。但还没有定义"什么算 durable"。如果边界不清，Outliner 会要么太脏，要么太空。
   建议先产出一张沉淀规则表：偏好 / 约束 / 主题 / 未完问题 / 任务 / 协作规则，各自的触发条件、默认节点形态、来源标记和撤销路径。

2. **跨会话检索的 scope 到底怎么切**
   用户心智已经偏向"全局个人历史"。现在剩下的张力不是"要不要全局"，而是：AI 自动引用时是否也默认全局，还是只有用户显式检索时走全局。
   建议把 scope 拆成两层讨论：用户主动检索的默认 scope；AI 自动引用的默认 scope。

3. **AI 什么时候该把旧东西拉回来**
   不是"能不能检索过去"，而是"什么时候引用过去"。过于积极会像翻旧账，过于保守又失去跨会话价值。
   建议定义三档策略：直接引用、仅内部参考、先问一句再引用，并为每档给出触发信号。

4. **用户控制与可解释性**
   现在已明确：来源默认收起，只在展开时显示。剩下要定义的是最小控制面：用户如何查看来源、排除某段历史、撤销 AI 刚写入的 durable memory、区分过程历史和正式记忆。
   建议把"可审查、可修正、可删除"拆成具体 UI 能力清单，而不是停留在理念层。

### P1：决定产品是否跑偏的问题

5. **Chat 作为主入口，会不会压扁结构思考**
   当前答案是"不会，因为 Outliner 是精修层"，但还需要继续验证。若用户长期只停留在 Chat，soma 会滑向聊天产品。
   建议继续讨论"从对话回到结构"的强引导机制，以及哪些高价值时刻必须把用户带回 Outliner。

6. **冷启动怎么过**
   这套 vision 的长期价值建立在节点、聊天和跨会话连续性之上，但 Day 1 用户没有这些积累。
   建议分开回答两个问题：Day 1 的即时价值是什么；连续使用 1 周后新增的复利价值是什么。

7. **对外到底卖什么**
   内部可以同时理解"浏览器里的 AI 思考伙伴 / 知识协作系统 / 认知产品"，但对外不能同时卖三件事。
   建议后续单独收敛一版对外叙事，只保留一个一级定位，其他都退到二级说明。

### P2：后续仍需深入的问题

- 认知摩擦的精细设计（AI 什么时候该问，什么时候该闭嘴）
- 节点编辑与模型上下文同步（`transformContext` 注入实时状态）
- AI 主动创建节点的边界和频率控制
- Taste 学习与摩擦的关系
- 从当前产品到 v2 形态的渐进式迁移路径
- slogan 更新（"Think with your AI" 或其他候选）
- Chat 中的任务面板设计（进度、结果、并行任务的呈现）
- Agent 编排架构（orchestrator、subagent、inter-agent 通信）
- `#task` 标签泛化（统一 Spark、异步调研、用户自定义任务的状态机制）

### 建议的讨论顺序

建议下一轮优先按这个顺序推进：

1. Chat → Node 的沉淀规则
2. AI 自动引用 past chats 的 scope 和触发时机
3. 用户控制与可解释性
4. Chat 与 Outliner 的导流关系
5. 冷启动价值
6. 对外定位
